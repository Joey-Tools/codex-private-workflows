# Personal Guidelines

- Keep this file short. Cross-repo rules belong here; repo-specific policy belongs in the repo `AGENTS.md` or a repo-local skill.
- Explanations, discussion, analysis, and summaries use Simplified Chinese. Code, comments, identifiers, commit messages, and Markdown code blocks use English.
- Do not claim a test, build, lint run, or command was executed unless it was actually run.
- If a final answer includes app directives such as `::git-stage`, `::git-commit`, `::git-push`, or `::git-create-pr`, include at least one short natural-language summary first; directive-only final answers are not allowed.
- For non-trivial logic changes, prefer adding or updating tests.
- Before proposing destructive commands or irreversible operations, explain the risk and name a safer alternative when one exists.
- Do not default into basic syntax or beginner-tutorial explanations unless the user asks for them.
- For shell script edits, run `bash -n` and `shellcheck` when available before commit; if `shellcheck` is unavailable, state that and report the fallback validation you actually ran.
- For non-trivial work, read the relevant code, docs, logs, or source material before proposing a concrete plan or making substantial edits.
- For moderate or complex tasks, separate planning from implementation: align on the approach first, then execute, and avoid unnecessary clarification loops when reasonable assumptions are enough.
- Do not expand the scope into unrelated rewrites or side quests. Low-level fixes for mistakes introduced in the current task stay in scope and should be corrected directly.
- Use `$joey-skill-authoring` when creating or revising skills, or when deciding whether guidance belongs in `AGENTS.md`, `SKILL.md`, or `references/`.
- Use `$agile-delivery-workflow` when Joey explicitly asks to prioritize an early usable MVP or says `先可用`, `快速迭代`, `偏 agile`, `MVP`, or `scout`; stop first at a focused checked signed local commit, then continue to normal delivery/PR workflow only when requested.
- Use `$codex-rules-hygiene` when auditing or pruning `~/.codex/rules/default.rules`, especially after recent approvals suggest wrapper drift, helper gaps, or approval-log buildup rather than genuinely new stable command families.
- Use `$codex-session-mining` when a task depends on `~/.codex/session_index.jsonl`, `history.jsonl`, or `sessions/**/rollout-*.jsonl`, such as recent-session audits, prior-work recovery, or mining repeated skill friction.
- Use `$change-delivery-workflow` for the local pre-commit delivery gate on non-trivial work: implement, build, test, update docs, run local/internal review, then commit. Use `$pr-readiness-review-workflow` after a feature is review-ready or PR-bound, when Joey asks the current agent to drive the named PR readiness review gates, CI/review waiting, merge-readiness, or `codex thread <session-ID>` review-comment repair. If Joey asks for the workflow to continue "before merge", "until merge-ready", or "在合并前停止", chain both skills: local commit is not the terminal state; the terminal state is merge-ready or a clear blocked report. Do not use this workflow for independent/review-only PR reviewer prompts that forbid PR orchestration, fixes, CI waiting, or starting reviewers; output findings only.
- For reviewable tasks, prefer a `wip/<topic>` branch plus fixed `base_sha..head_sha` review ranges over live working-tree review; append fixes on the `wip/` branch, then squash onto the target branch after tests and the final whole-range review/gate. Use `~/.codex/bin/codex-wip-branch` and `~/.codex/bin/codex-squash-merge-wip` when they fit.
- For internal review, default to the helper-backed root `codex-review` lane via `~/.codex/skills/review-orchestration-playbook/scripts/isolated_review stateful <start|status|wait|final> --entrypoint codex-review`, because it is the default agentic lane and preserves the historical single-lane final artifact contract. Use `codex-readonly` as the deterministic fallback when `codex-review` is unavailable, inconclusive, or when you intentionally need an exact diff-fed findings baseline, and default that fallback to the same helper `stateful <start|status|wait|final>` lifecycle rather than a plain one-shot run. Reserve direct one-shot `codex-readonly` only for narrow local smoke/debug cases where no durable final artifact is required and manual interruption risk is acceptable. Use `codex-parallel` only when you explicitly want dual-lane coverage and the caller can consume an aggregate `stateful final`; it is not the default path. Keep the custom `reviewer` agent in `~/.codex/agents/reviewer.toml` only as an explicit weaker fallback when that helper path is unavailable. Do not route review through a default coding subagent, and do not substitute Cursor/headless `agent` CLI for the Codex internal-review lane.
- Use `$review-orchestration-playbook` when Joey explicitly asks for OpenCode, Cursor Agent, Copilot, Claude, helper-backed internal review, fresh-context/subagent code review, or equivalent lane orchestration. Non-Codex external reviewers are opt-in; PR readiness still requires the `independent-codex-pr-review` gate owned by `$pr-readiness-review-workflow`. For review-only child prompts that forbid starting other reviewers, do not orchestrate another lane; output findings only.
- Use `$remote-host-context` when task evidence may span Joey's local machine, `miku-bot-dev`, or `hoteng-srv-01`, especially for work-report audits, session/history scans, or recent-activity recovery that would be skewed by local-only evidence.
- For workflows rooted in `~/.codex`, `~/.codex/rules`, `~/.cursor/cli-config.json`, Apple Notes, downloaded local archives, or other host-level machine state, prefer a personal skill in `~/.codex/skills` over a repo-local mirror unless the workflow genuinely depends on repo-specific scripts or policy.
- Unless a repo-local `AGENTS.md`, repo-local skill, or stronger project convention says otherwise, use `$project-journal` as the default Joey repo tracking workflow: write ordinary PR/thread/workstream state to `docs/project_journal/YYYY/MM/*.md`, keep `docs/PROJECT_STATE.md` and `docs/PROJECT_TODO.md` as stable short entrypoints for repo-wide status and cross-workstream backlog only, treat generated `docs/project_journal/INDEX.md` as a local ignored artifact, and avoid maintaining giant tracked status/TODO files.
- For PR-bound project journal updates, first check the repo's merge model. In squash-merge repos, write tracked journal docs as the target-branch post-merge state and prefer updating the relevant per-workstream journal over the top-level entrypoints: if the PR fully completes the workstream, use `status: completed` before merge and keep transient states such as "waiting for merge", "not merged yet", or "ready for review" in the PR body, checklist, or comments. If the merge model is unclear or not squash-merge-only, keep each commit's docs self-consistent and do not mark work completed before the same commit contains the completed implementation.
- Use `$bug-triage-playbook` for log-driven debugging, regression analysis, flaky behavior, or root-cause work that starts from symptoms.
- For repeated Jenkins or remote archive triage, keep auth/fetch preflight direct and use the `bug-triage-playbook` helper script/reference for the repetitive local archive-inspection steps instead of rebuilding the same shell wrapper chain from scratch.
- When the user provides a specific remote log, archive, crash report, or build URL, treat it as the primary evidence. Verify access and auth first; do not quietly fall back to stale local artifacts.
- When Cursor Agent, a legacy review CLI, or a remote artifact fetch may need approval, network, or auth, do a narrow preflight first and report the exact missing gate before attempting the full workflow.
- When an approved command family likely already exists, prefer invoking the real executable directly instead of wrapping it in `bash -lc` or `/bin/zsh -lc`. Use a shell wrapper only when pipes, redirection, heredocs, or inline env assignment are actually needed.
- When a review or debugging pass needs adjacent context beyond the explicitly scoped files, verify the candidate path first with `rg --files`, `git diff --name-only`, or a similarly bounded listing before opening it; do not guess sibling filenames and probe them with `sed`/`cat`.
- Before broad `rg`/searches in large repos or generated-output-heavy trees, bound the result set in the command itself: first scope candidate directories or add excludes such as `out/`, `build/`, `vendor/`, archives, and generated artifacts; prefer `rg --files` or `rg -l` to identify candidate files, then open only a small explicit selection with narrower globs. Treat `max_output_tokens` only as a display backstop, and do not treat per-file limits such as `rg -m` / `--max-count` as a total-output cap.
- When searching for reusable context across sibling repositories or worktrees, do not point raw `rg -n` at broad parent directories such as `/Users/hoteng/Program/GitHub` or at multiple repository roots. First choose the exact repo/worktree and an explicit file list; exclude dependency lockfiles such as `pnpm-lock.yaml`, `package-lock.json`, and `Cargo.lock` plus generated artifacts unless the task is specifically about dependency metadata.
- For high-frequency identifiers, large alternation patterns, or searches that mix source, tests, docs, logs, or `~/.codex` trees, start with `rg -l`, `rg --count`, or a bounded file list; do not run raw `rg -n` until the candidate set is small. For large review ranges, avoid `git diff --unified=<large>` across the whole diff; inspect `--stat` / `--name-only` first, then diff selected files. For process diagnostics, prefer PID/name-scoped probes such as `pgrep -af <pattern>`, `ps -p <pid>`, or `lsof -nP -p <pid>` over full `ps aux` / `ps -ef` dumps.
- For installed package metadata, package-manager cache JSON, lockfiles, or binaries, check file type and size before searching; use structured parsers or bounded extraction pipelines that emit only candidate keys/symbols, not raw `rg -n` over large single files or unbounded `strings` output.
- For verbose build/test commands that can stream huge logs, especially `xcodebuild` or Swift builds with many warnings, keep the full log in a task-scoped file and surface only targeted failure lines or a short tail; do not repeatedly poll noisy sessions with high output limits once the output is already broad.
- Execute single-command instructions directly, even when they need escalation, instead of writing a script file first. When a task can be cleanly split into a few direct commands, run those commands one by one instead of wrapping them in a multi-step script. Reserve HERE DOC for short read-only probes that stay inside the current sandbox and do not need approval. For non-trivial multi-line shell or Python logic, do not inline it via HERE DOC or `python -c`; prefer writing a task-scoped script file and executing it instead. Before requesting approval for that script, briefly state what it will do.
- When passing URLs or other arguments that contain shell metacharacters such as `*`, `?`, `[`, `]`, `&`, backticks, `$`, or newlines, prefer direct argv forms; if a shell wrapper is unavoidable, quote those arguments explicitly so the shell cannot rewrite them before the helper sees them. For GitHub PR/issue bodies or other Markdown payloads, write a task-scoped file and use tool-native file input such as `gh pr create --body-file` instead of inline shell `--body`.
- In sandboxed or approval-gated sessions, if `git rev-parse --git-dir` and `--git-common-dir` differ or point outside the cwd repo, treat it as a linked-worktree/shared-gitdir write path. Not only `git switch`, `git worktree add`, and `git submodule update --init`, but also `merge`, `stash`, `restore`, `cherry-pick`, branch/ref updates, and similar writes can fail on `.git/worktrees/.../*.lock`, `.git/index.lock`, `.git/sequencer`, or `.git/config`; preflight that split first and move to a writable source-repo context or narrow approval instead of repeating the same doomed git write.
- For cleanup inside a repo's `.codex-tmp`, prefer `~/.codex/bin/codex-clean-tmp <subpath...>` or `~/.codex/bin/codex-clean-tmp --all` instead of raw `rm -rf`; treat that helper as protected infrastructure and do not modify it unless Joey explicitly asks.
- For recurring scheduled/background chores, prefer Codex Automations paired with skills; use Rules to control sandbox escapes instead of treating `AGENTS.md` as a pseudo-hook system.
- For `lsof`, prefer the normalized `-nP` forms such as `lsof -nP -p <pid>` or `lsof -nP -iTCP` so diagnostics and recurring approvals do not drift across `-n` / `-nP` variants.
- When creating substantial temporary artifacts (for example under `/tmp` or `.codex-tmp`), prefer task-scoped directories, clean them up before finishing when safe, and explicitly report any leftovers you intentionally keep.
- If helper validation scripts fail because local dependencies are missing, do a lightweight fallback validation and state exactly what was checked.
- If a repo-local `AGENTS.md` has unrelated user edits, mention it once and continue unless it directly conflicts with the current task.
- When an external review path or live validation gate stays inconclusive after bounded retries, stop at a crisp decision point with evidence instead of looping indefinitely.
- For any long-running command you may need to poll, interrupt, or harvest final output from, start it with a pollable TTY/PTY shape; do not assume `write_stdin` can attach to a plain-pipe session after stdin closes. For external review helpers, if the subprocess is still alive but output goes dark, change the session shape before launching a second reviewer.
- Prefer `gh` CLI for GitHub-specific interactions.
- For debugging or review tasks, lead with the decisive evidence or findings rather than narrating the whole search.
- When only key evidence matters, quote or summarize the most relevant 5-10 log lines and label them as key lines instead of pasting whole logs.
- When build or tooling details matter, include the relevant tool path and version to reduce follow-up questions.
- For multi-line commit messages, use `git commit -F` or multiple `-m` flags instead of embedding literal `\n`.
- Do not suggest history-rewriting Git commands such as `git rebase`, `git reset --hard`, or `git push --force` unless explicitly requested.
