# PR Readiness

Use this reference after the local delivery gate has produced a reviewable commit and the parent request owns PR creation/update, review/CI follow-up, or merge-readiness reporting.

## Authorization

- Confirm repository owner/name, base branch, head repository/branch, draft/ready state, current head commit, dirty state, and merge model.
- Joey-owned/default-authorized repositories may be pushed and opened/updated when the parent request asks for a PR, full workflow, merge-ready, stop-before-merge, or triple review.
- For any other target, stop and request explicit confirmation listing the exact repository, base, head, and draft/ready state.
- These phrases authorize PR creation/update when the target check passes; they never authorize merge.

## Gate Sequence

- Measure supervisor ownership-handoff, checkout, reviewer-launch, and reviewer duration limits with monotonic clocks; wall-clock changes never extend them.
1. Establish or reuse the PR and read metadata, review threads, required checks, rulesets, and branch/base state with the bounded probes in [github-pr-probes.md](github-pr-probes.md).
2. Record the current PR head and freeze the local scope as `<merge_base>..<head_sha>`.
3. Run `offline-frozen-diff-review` first through the stateful pinned Codex helper on the exact frozen range with `--keep-workspace`. Require its retained `preflight.json`, companion `control-artifact-state.json`, and `review.diff` before launching any separate Codex session. The preflight must contain a scanned `primary_diff` attestation whose relative path, byte length, and SHA-256 exactly match that retained `review.diff`; bind those fields, the retained source attempt/path, and the exact `<merge_base>..<head_sha>` range, and reject a diff larger than 128 MiB. Bind the companion control-state SHA-256 for source custody, but do not substitute it for the `primary_diff` attestation. This proves the frozen workspace, diff, and prompt passed the sensitive-content and escaping-symlink checks and supplies the primary evidence that the independent lane will seal into its own checkout. `--keep-workspace` provides initial availability, while an actual shared BSD `flock` lease on that state directory's `cleanup.lock` plus an already-open no-follow source descriptor provides race-safe custody: explicit `stateful cleanup` uses the exclusive side of the same lock and cannot unlink the workspace until destination sealing releases the lease. Do not intentionally request cleanup earlier. If the reviewer runtime becomes unavailable only after that preflight, use the helper-retained frozen workspace for the configured clean-context `reviewer` fallback with the same evidence contract, collect the artifact, and retain the same source custody until the independent copy no longer needs it. If the helper cannot complete the preflight itself, stop instead of bypassing it.

### Raw Git LFS Reference-Compatible Pointer Gate

The materializer knows every regular blob's declared size before consuming its payload. Independently of current-head attributes, it must exact-buffer every **nonempty** Git regular-file blob whose declared raw blob size is strictly less than 1024 bytes in at most 1023 bytes and locally apply a `git-lfs` 3.7.1 `lfs.DecodePointer`-equivalent, content-only recognizer. Empty blobs are harmless/pass-through; blobs of at least 1024 bytes are outside this content gate. The recognizer trims outer whitespace, uses Go `bufio.ScanLines` behavior, ignores only zero-length `ScanLines` records, splits every remaining line at the first ASCII space, and rejects an internal whitespace-only line rather than treating it as blank. It accepts the exact v1 aliases `http://git-media.io/v/2`, `https://hawser.github.com/spec/v1`, and `https://git-lfs.github.com/spec/v1`. Core keys must advance in `version` -> `oid` -> `size` order, but reference-valid extensions may appear before or between those core keys and may be unsorted; extensions after `size` are rejected. Validate the main and extension OIDs as exact `sha256:` plus 64 lowercase hexadecimal digits, parse size exactly as a nonnegative base-10 signed-64-bit integer (including upstream-accepted `+0001`, leading zeros, and `-0`), and reproduce upstream extension-key/prefix, priority, duplicate-key overwrite, duplicate-priority, and sorting semantics rather than replacing them with a stricter canonical regex. Therefore outer whitespace, CRLF, true zero-length records, missing final LF, either legacy alias, and reference-accepted noncanonical extension placement/order still block. Any nonempty accepted pointer is prelaunch `blocked-checkout-lfs-pointer` with `review_status=not-run`; invalid version/OID/size/extension near-misses pass this detector only after exact local rejection. A PR cannot bypass it by deleting or rewriting `.gitattributes`. Detection is offline and reads only the already streamed raw blob: do not invoke `git lfs`, consult an LFS object store, download LFS content, perform lazy fetch, or access the network. Failure to complete the required exact check is `inconclusive`; the frozen-head attribute query remains an independent second gate, neither gate substitutes for the other, and attributes are never the sole LFS-pointer detector.
