# Review Lane Contracts

Use this contract for helper-backed review, a clean-context `reviewer` fallback, or a findings-only child.

## Scope And Evidence Budget

- Bind the review to one frozen `base_sha..head_sha` range or an explicit diff artifact.
- Use the supplied diff as the primary surface; read nearby tracked files only to validate a concrete concern.
- Count before printing changed-file lists, diff headers, `--stat` / `--numstat`, or large search results; cap first-stage samples.
- Treat line-producing `rg -n`, including `rg -n -C`, as a second-stage read against one exact file, hunk, or symbol window after `rg -l` / `rg --count`.
- Do not start with `git diff --unified=30/40/50/60/80`, `git diff --function-context`, `git diff -W`, low-context multi-file diffs, bare `cat`, whole-file `nl -ba`, bare `git show <rev>:<path>`, broad multi-file `rg -n`, or full untracked inventories.
- Before every tool call, rewrite a forbidden broad shape into a count probe, one hunk, one exact symbol, or a narrow `sed` window.
- After an 800+ line or 10k+ token result, narrow the next read.
- Do not run broad builds/tests/package-manager commands from the review lane. Use a small read-only probe only when necessary to validate a finding.

## Output

- Return findings only, ordered by severity, with file/line references when possible.
- Report correctness, security, data-loss, behavioral regression, performance/resource, reliability, and missing-test risks.
- Skip style-only, naming-only, formatting-only, and speculative comments.
- If there are no actionable findings, reply exactly `No findings.`
- Intermediate reasoning, file reads, progress, and keepalives are not final artifacts.

### Git LFS Reference-Compatible Pointer Content Gate

- The raw materializer already knows each regular blob's declared size from the authenticated `cat-file --batch` header. Independently of every attribute result, for every **nonempty** Git regular-file blob whose declared raw blob size is strictly less than 1024 bytes, exact-read the complete payload into a buffer capped at 1023 bytes and run a local content-only detector equivalent to `git-lfs` 3.7.1 `lfs.DecodePointer` acceptance before copying it into the placeholder. The empty blob is explicitly harmless/pass-through even though upstream represents it as `EmptyPointer`; it must not produce `blocked-checkout-lfs-pointer`. For a nonempty candidate, trim outer whitespace exactly as upstream `bytes.TrimSpace`, scan lines with Go `bufio.ScanLines` semantics (therefore accept CRLF and a missing final LF), ignore only zero-length `ScanLines` records, and split each remaining line at its first ASCII space. An internal whitespace-only line is nonempty and rejects; it is not an ignorable blank line. Accept exactly the three upstream v1 version aliases `http://git-media.io/v/2`, `https://hawser.github.com/spec/v1`, and `https://git-lfs.github.com/spec/v1`.
- Mirror upstream key/value validation rather than requiring canonical re-encoding. Require core keys to advance exactly in `version` -> `oid` -> `size` order, while reference-accepted extension lines may occur before or between core keys and need not arrive in priority order; a nonblank extension after `size` is rejected. Require the main OID and every extension OID to be exactly `sha256:` followed by 64 lowercase hexadecimal digits. Parse `size` exactly as a nonnegative base-10 signed-64-bit integer, including upstream-accepted `+0001`, leading zeros, and `-0`, while rejecting a negative result or overflow. Reproduce upstream's extension regex-prefix and parse semantics—one decimal priority digit `0..9`, a word-character name prefix, the remainder after the second hyphen as the parsed name, same-key map overwrite, exact SHA-256 extension OID, rejection of different extension keys with the same parsed priority, and priority sorting—rather than substituting a stricter canonical regex. Outer whitespace, CRLF, blank lines, a missing final LF, a legacy version alias, and unsorted or interleaved reference-valid extensions make a pointer noncanonical but do not make it safe to expose. Any nonempty payload accepted by those rules aborts prelaunch as `blocked-checkout-lfs-pointer` with `review_status=not-run`; never copy or expose it as usable reviewer evidence. Invalid version/OID/size/extension near-misses continue only after the complete exact local parse rejects them. Any inability to complete this required exact check is prelaunch `inconclusive`, never a downgrade to attribute-only handling.
- Do not infer a non-pointer from current-head attributes: a PR that deletes or rewrites `.gitattributes` cannot bypass the content gate. The existing frozen-head attribute query remains a second independent gate for paths whose conversion metadata survives, and neither gate substitutes for the other. This detector reads only the already streamed raw Git blob and must not invoke `git lfs`, consult an LFS object store, download LFS content, perform a lazy object fetch, or access the network. Blobs at least 1024 bytes, the empty blob, and nonempty small blobs rejected by the reference-compatible parser continue through the existing bounded raw-copy, independent attribute, and integrity gates.
