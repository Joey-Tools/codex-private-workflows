# Isolated Review Helper Contract

The canonical helper is:

```bash
$HOME/.codex/skills/review-orchestration-playbook/scripts/isolated_review
```

It runs one low-level supplied-diff reviewer against one frozen Git range. It is retained for compatibility, helper-security maintenance, and targeted runtime debugging. It is not the canonical implementation of single, double, or triple review; the parent skill composes named lanes with separate clean Git worktrees and no prepared reviewer diff.

## CLI

Legacy helper Codex foreground invocation (does not satisfy canonical single review):

```bash
isolated_review \
  --repo /path/to/repo \
  --reviewer codex \
  --base-ref <base_sha> \
  --head-ref <head_sha>
```

Stateful:

```bash
isolated_review stateful start \
  --repo /path/to/repo \
  --reviewer claude \
  --egress-consent explicit-claude-review \
  --base-ref <base_sha> \
  --head-ref <head_sha>

isolated_review stateful status --state-dir <state_dir>
isolated_review stateful wait --state-dir <state_dir>
isolated_review stateful final --state-dir <state_dir>
isolated_review stateful admission --state-dir <state_dir>
isolated_review stateful cleanup --state-dir <state_dir>
```

Synthetic-token catalog inspection is read-only:

```bash
isolated_review synthetic-tokens validate
isolated_review synthetic-tokens list --json
isolated_review synthetic-tokens get <id> --json
isolated_review synthetic-tokens list-exemptions --json
isolated_review synthetic-tokens audit-master --repo <path> --ref <sha> --exemption <id>
```

Review commands retain repeatable `--synthetic-secret-exemption <id>` only as deprecated compatibility syntax. Known IDs may still be validated, but selection must not change the scanner, counter, path rules, or admission outcome. Read [synthetic-token-fixtures.md](synthetic-token-fixtures.md) before changing the catalog or removing this compatibility surface.

Always pass `--state-dir`; it is not positional. `stateful wait --timeout-seconds` accepts only a non-negative finite value, bounds the caller's wait, and does not kill or downgrade a healthy reviewer. `stateful cleanup` explicitly removes a workspace retained by `--keep-workspace` or for helper-specific recovery while preserving state artifacts. A retained `.git`-free helper workspace must never be handed to the canonical fresh-context Codex lane.

Canonical PR/master admission and `merge-ready` use the separate direct `isolated_review secret-admission` command, which starts no reviewer. This helper's `stateful final` / `stateful admission` pair is retained only when a low-level helper run was independently requested: collect final first, then query admission against that same state and current head. Those helper results remain independent—final may succeed when helper-state admission reports violations or an inconclusive bounded audit. Helper-state admission exit `0` means `clean`, exit `1` violations, exit `3` pending, and exit `75` inconclusive; none makes this helper a required PR-readiness producer. Foreground mode produces no helper-state admission evidence. A head change invalidates both helper results. Secret admission never delays, suppresses, redacts, or gates reviewer launch; the reviewer continues to receive the original consented tracked scope.

The parent acquires an exclusive runner lock at the first preparation cleanup handoff, before any helper-private secret bytes are written, and passes that same file descriptor to the child for the child's full lifetime. It also passes the selected reviewer and exact optional egress-consent value as child argv. Terminal `_run-state` validates the inherited lock first, requires the path-loaded state to match both trusted argv values exactly, and launches only with those argv-bound values. Replacing `state.json` therefore cannot switch a Codex-only run into Claude or Copilot egress. Cross-process `status` / `wait` trusts the lock, not PID existence, so a reused PID cannot masquerade as the review runner or race cleanup during preparation.

## Low-Level Helper Reviewers

`--reviewer codex`:

1. `gpt-5.6-sol`, `xhigh`
2. `gpt-5.5`, `xhigh`, only after explicit model entitlement/policy denial

This is the legacy supplied-diff Codex runtime. It does not load the normal instruction stack or expose the clean Git workspace required by canonical single review, so its result cannot satisfy a named review shape.

`--reviewer claude`:

1. Claude Code `claude-opus-4-8`, `max`
2. Claude Code `claude-opus-4-7`, `max`, only after explicit organization-policy or model-entitlement denial for Opus 4.8
3. Copilot CLI `claude-opus-4.8`, `max`, only when the secure Claude Code runtime is deterministically absent/unavailable or both Claude Code Opus models are entitlement-blocked; authentication failure never reaches this step
4. Copilot CLI `claude-opus-4.7`, `max`, only after the same explicit organization-policy or model-entitlement denial for Opus 4.8

This helper lane also uses a `.git`-free supplied-diff workspace. Even when the actual runtime is Claude Code, it does not satisfy the canonical double/triple Claude lane until the clean-Git/no-prepared-diff contract is met. Any Copilot result is a low-level fallback result only and never counts as Claude Code.

Claude Code CLI releases are not pinned to one current patch. The Claude runtime runs in safe mode and accepts publisher-verified releases `>=2.1.211,<3.0.0` only when the detected platform, outer sandbox, credentials, launch flags, safe-mode semantics, and structured output contract pass. The `2.1.211` floor trusts current credential-store coordination, refresh behavior, and explicit login-expiry signals; it replaces the old helper-managed freshness warmup without changing the signed-manifest and capability-range model. Codex CLI and Copilot CLI likewise have no exact executable-version pin; their existing identity, capability, and output verification remains authoritative. These CLI policies do not change the explicit reviewer-model policy above: model IDs remain intentionally pinned.

The Copilot IDs follow GitHub's authoritative [supported models matrix](https://docs.github.com/en/copilot/reference/ai-models/supported-models), which lists Opus 4.8 and Opus 4.7 for Copilot CLI. The shorter command-reference example table can lag product availability and is not treated as a runtime allowlist; exact model verification still fails closed if the installed CLI rejects or substitutes any ID.

The low-level Claude helper requires either `--egress-consent explicit-claude-review` or `--egress-consent explicit-claude-with-copilot-fallback`. The helper saves this value in state and writes `egress.json`; it refuses to start the external lane without it. `explicit-claude-review` authorizes Anthropic only. `explicit-claude-with-copilot-fallback` is valid only after Joey separately requests and authorizes both the Anthropic review and this helper's compatibility GitHub Copilot fallback. Named single/double/triple phrases are not helper consent markers. No Copilot artifact satisfies the required Claude Code lane, and no consent mode turns an authentication problem into fallback eligibility.

Every new state and `egress.json` record is machine-labeled `review_contract: supplied-diff-no-git` and `named_lane_eligible: false`. `stateful status` exposes those fields, while `attempts[].runtime` remains the authoritative actual backend. Consumers must not infer a named lane from the top-level requested helper reviewer, exit `0`, or the findings-only output of `stateful final`; that command intentionally returns the saved low-level artifact, not a named-lane envelope.

The foreground compatibility command likewise prints only the raw helper artifact and does not emit a machine envelope. Its command surface is unconditionally ineligible for named-lane counting. Automation that needs machine-readable contract metadata must use `stateful status`; it must never ingest foreground stdout or `stateful final` as named-review evidence.

Model verification normalizes punctuation only and then requires exact equality. A requested `gpt-5.5` never accepts `gpt-5.5-mini`, `gpt-5.5-codex`, or any other suffix as the same model.

Capacity, overload, rate limits, timeouts, network/5xx errors, missing artifacts, silent model substitution, and review findings never trigger a model downgrade. The helper records every attempt and reports transient failures without switching models.

Fallback classification uses stderr plus explicit structured CLI error events and error-schema fields only. Partial `result`/`data` payloads, reviewer tool output, and repository text on stdout are never scanned for entitlement or transient substrings.

For Claude runtime preparation, deterministic automatic absence, unsupported platform enforcement, or a cleanly missing non-security capability from an automatically discovered candidate may be eligible for the already-authorized Copilot fallback. Publisher fingerprint/signature/checksum failures, contradictory safe-mode claims, outer-sandbox mismatches, unsafe runtime state, and invalid explicit overrides are blocked security or configuration failures and never authorize fallback. An explicit `CODEX_REVIEW_CLAUDE_PATH` that lacks the required GPG verifier, outer probe sandbox, or trusted review-tool prerequisite is also blocked rather than reclassified as automatic absence. Probe, manifest-download, executable resolve/stat I/O, file-race, and bounded-I/O failures are inconclusive and likewise never authorize fallback.

Authentication is a pause boundary, not runtime availability. Access-token expiry alone is not login expiry and does not block launch when a valid refresh token is present. A selected local credential that is missing, malformed, unsafe, lacks a usable refresh token, or is actually rejected by Claude as `Login expired`, HTTP 401, or refresh failure produces `blocked-authentication` plus the action `claude auth login`. A rejected explicit API key produces the same terminal classification but directs the operator to unset or replace `ANTHROPIC_API_KEY`; an API key is never evaluated for an OAuth refresh token. The helper stops until Joey explicitly retries. This classification never authorizes Copilot under any egress-consent mode and never fabricates a Claude model attempt when launch did not occur. A structured authentication error from a launched review has the same terminal classification even if model metadata is otherwise verifiable. When that rejection coincides with post-attempt refresh-persistence uncertainty, authentication remains primary while the runtime report and operator-facing error retain the validated recovery carrier, artifact, cleanup scope, and persistence diagnostic; a report-write failure inherits the same recovery metadata instead of obscuring it. Model entitlement remains distinct: a strict final-review error with trustworthy requested/effective-model evidence may advance through the pinned Claude model chain and, after both Claude models are entitlement-blocked, to an already-consented Copilot backend.

Credential-source read supervision, loopback setup, and broker construction failures remain fail-closed. A deterministic absence of the secure broker implementation may be secure-runtime unavailability; an unsafe broker or credential source is blocked. When the broker source and compiler are present, a compiler start/I/O failure or nonzero build is inconclusive and never authorizes fallback. Across the Keychain broker and both TCP and Unix CONNECT proxies, only an explicit OS policy or socket-capability bind errno is deterministic runtime unavailability. An unknown, resource/capacity, or address-contention bind error, Unix-socket permissioning failure, thread-start failure, serve-start uncertainty, post-ready serve-loop failure, timeout, output-limit breach, drain failure, process leak, or file race is inconclusive. None of these conditions is reclassified as an authentication denial merely to enable fallback.

Local-login refresh persistence is an attempt-time boundary, not deferred best-effort cleanup. On macOS, capability-broker handlers reject token strings that cannot encode as UTF-8 and validate every rotation before any filesystem work. Every structurally valid `W` generation admitted to durable staging atomically reserves attempt-scoped durable-journal quota before any filesystem write. The journal is bounded to eight generations and 8 MiB of payload; its last generation and 1 MiB are reserved for terminal recovery, and each reservation consumes quota for the rest of the attempt even when the subsequent write or publication fails. The generation then commits to a distinct helper-private recovery carrier: the credential file, containing directory, new carrier-directory entries, review-container entry, review-root entry, and final carrier rename are synchronized and the exact payload is read back. Each Darwin synchronization performs `fsync` followed by `F_FULLFSYNC`; an unavailable or failed full sync NACKs the generation. If rename completed first, only exact re-proof of the acknowledged path and payload digest may retain the generation as recovery-only evidence; it is never a host-writeback candidate. Durable commit alone does not authorize success. At the broker's generation linearization point, the handler holds the current-generation lock, rechecks that its generation is still current and the runtime is not abandoned, and only then publishes and acknowledges it. When an update would consume the terminal reserve, the server atomically claims it only for the current pending generation, permanently closes later `W` admission, commits that exact payload into the final slot, and NACKs without publication or host writeback. A superseded candidate consumes no quota or filesystem work, and later requests receive explicit NACKs before callback execution. The terminal generation invalidates any older staged host-writeback candidate, marks credential inspection inconclusive, and pauses without Copilot fallback. Superseded and failed carriers remain in the bounded journal until post-quiescence finalization or failure recovery. Acknowledgement proves successful full-sync calls plus exact recovery staging, not host persistence or an absolute power-loss guarantee; `F_FULLFSYNC` is Darwin's strongest available best-effort barrier and storage hardware may still fail to honor it. After the broker server and every handler quiesce, a single outer-runtime owner re-reads and exact-verifies the newest carrier before removing older journal entries, then persists only the latest acknowledged credential under the artifact-certified primary and legacy locks; Keychain and the `pwd`-home file are synchronized only when their refresh-token fingerprints identify the same logical login. The final recovery carrier is removed only after verified host commit with the same Darwin full-sync boundary on file and parent-directory cleanup. A control-flow interruption while removing an older durable generation stops immediately before host writeback or latest-carrier cleanup. On Linux/WSL2, the verified runtime rejects non-UTF-8-encodable token strings and receives a helper-owned writable `/auth` carrier with config at `/auth/config`, so both the primary config lock and legacy `/auth/config.lock` sibling can be created. Before reading the host credential, the helper opens and retains a no-follow directory-descriptor chain from `/` through the credential parent; every source read, lock, atomic replacement, and directory sync stays relative to that chain, while any symlink ancestor, leaf symlink, edge replacement, or ownership/mode change pauses inspection without fallback. Borrowed refresh-lock anchors are descriptor-only, and the credential owner revalidates the complete chain at commit boundaries. Descriptor-bound cleanup failure never publishes the now-untrusted lexical path as a helper-owned stale lock; it reports that no authoritative pathname is available and pauses for independent identity-safe recovery. A bounded watcher persists each stable staged rotation without holding staged and host locks together, then a synchronous final drain runs after an ordinary watcher stop and before carrier cleanup. Stop linearizes closure of new background-writeback admission before the bounded join, so a candidate whose last blocking read completes after closure cannot begin host writeback. If the initial bounded join times out, the attempt becomes inspection-inconclusive and immediately retains the carrier; the context skips concurrent final drain, payload scrubbing, and cleanup, records the recovery path, and returns or re-raises a deferred control-flow signal without an unbounded wait. The timeout handoff uses an explicit worker-cleanup state transition, so the last side to complete the handoff closes the descriptor chain without waiting behind filesystem inspection. If a background host writeback was admitted before stop and remains in flight, the diagnostic explicitly marks host state ambiguous because that already-started commit may still complete; it never claims an exclusive recovery handoff. The watcher is daemonized only so an uninterruptible filesystem operation cannot keep the helper alive after that bounded failure report; normal paths still join it. If the watcher has joined but Claude left staged locks around a rotated credential, only a normal supervisor return may attest that the sandbox process group is quiescent. Under that proof, the helper preflights the exact primary and legacy staged lock paths as empty current-user `0700` directories, removes them legacy-first, and retries final persistence once. Timeout, process-leak, unsafe-lock, retry, or host-write uncertainty retains the private recovery carrier in the review container, records its path without credential contents, and pauses. Stale shared host locks remain inspection-inconclusive and require controlled cleanup after confirming no writer remains; the helper never reclaims them automatically. Any writeback, lock, carrier-race, or watcher uncertainty pauses without a login prompt or Copilot fallback.

The latest exact-verified macOS generation is always the canonical current carrier. When quota rejection, a malformed update, or durable-stage failure leaves post-quiescence finalization without a staged host-writeback candidate, the owner reports that carrier as the sole current `authentication.recovery_artifact`. It removes every other non-authoritative complete journal entry and reports a failed stale-entry cleanup separately as `authentication.recovery_cleanup_artifact`. If that newest generation cannot be re-proven, the owner claims no current carrier, preserves every generation, and reports only the stable recovery-root cleanup scope. If staged or non-staged control flow leaves any stale carrier unvisited, cleanup reporting is promoted to the macOS recovery-root scope without changing a still-proven canonical current carrier.

On macOS, the helper prevalidates its stable recovery-root scope before exposing the broker. Unquiescent shutdown first irreversibly closes the server's pending-update publication/ACK gate, then boundedly drains handlers already inside the commit critical section. Runtime abandonment latching and pending detachment are independent recorded states, so detachment may be retried without reopening publication. The abandonment callback is event-only and itself deadline-bounded; it never waits for the runtime-state lock. A terminal durable stage racing that event either self-registers or is transferred to the quiescence stage by the already-bounded recovery worker under the runtime lock. Recovery runs for a detached `None` payload as well as a credential payload and must converge that stage. Timed detach uses one deadline across the initial-credential and pending-update locks; a timeout transfers no payload and leaves ownership with the server. Recovery-timeout reporting never waits for the runtime-state lock: a nonblocking snapshot may report an exact proven candidate, while lock contention conservatively reports only the stable macOS recovery-root cleanup scope. Once quiescence is unproven, outer finalization does not wait for or consume runtime state still owned by a late handler or recovery worker. Whenever abandonment recovery observes any durable journal entry or quiescence stage, its final error also reports that recovery root so old generations, a failed quiescence stage, and any fallback retain visible ownership even beside an exact current carrier. After an actual bounded recovery timeout, only the handler/recovery state machine owns in-flight stage cleanup; any late stage or fallback is exact-cleaned by that owner or remains inside the pre-reported root scope. When broker-server and handler quiescence cannot be proven, that handler/recovery state machine retains the latest exact-verified durable carrier and may durably materialize a newer pending structurally valid payload; outer finalization does neither. After proven quiescence, an outer-runtime owner whose complete guarded host writeback fails retains the latest exact-verified durable carrier and may durably replace it with a newer proven pending payload. This post-quiescence case includes Keychain-only bounded retry exhaustion, a first file-carrier write failure, and a partial same-login dual-carrier commit. Each carrier and its `config` directory are current-user `0700`, `config/.credentials.json` is a single-link regular file with exact mode `0600`, and no carrier path is granted to the Claude Seatbelt profile or passed as a child environment value. The carrier is not a new host credential source: the helper reports only an absolute path whose exact payload was created and verified when claiming a current recovery credential, never credential contents, and pauses as inspection-inconclusive without a login prompt or Copilot fallback. An unverified or incomplete file is cleanup-only and may be reported only through `authentication.recovery_cleanup_artifact`. An uncertain shutdown with no `W` generation, or a failed carrier creation that left no verifiable artifact, preserves the shutdown or persistence error without setting current recovery-credential metadata; pre-reported root cleanup metadata remains separate. If recovery replacement or finalization leaves an exact complete recovery update artifact, the helper retains and reports that exact artifact only after post-failure no-follow path-identity and payload verification; a later successful, read-back-verified replacement removes bounded stale recovery update artifacts. An actually existing incomplete update that cannot be removed or safely inspected is cleanup-only and must never be presented as a current recovery credential; a temp that disappears before final verification and a merely precomputed or attempted path are not recovery metadata.

When no `W` generation or verified artifact exists, the helper sets no current `authentication.recovery_artifact`; a pre-reported recovery-root `authentication.recovery_cleanup_artifact` remains permissible cleanup metadata.

Recovery reporting distinguishes the current credential from cleanup residue. Before a verified recovery commit, only a complete uncommitted update whose no-follow path identity and exact payload are reverified after the triggering failure is `authentication.recovery_artifact`. Once a replacement has been fully fsynced and read-back verified at `config/.credentials.json`, that verified file is the current `authentication.recovery_artifact`. Recovery-carrier removal advances cleanup ownership only after each destructive syscall from the credential file to `config`, the carrier, and finally the stable recovery root. Cleanup metadata must name an existing no-follow identity-stable scope: a vanished child is never reported, while entry churn beneath the same directory inode does not invalidate that directory as the cleanup scope. If deletion of an older temp fails, or an unverified or incomplete temp cannot be removed or inspected, report its exact path separately as `authentication.recovery_cleanup_artifact`; never label an older, unverified, or incomplete artifact as newest or current. If the temp disappears before final post-failure path verification, report neither field for that path.

Current recovery-artifact publication is source-content-bound rather than path-derived. The caller supplies the digest from the already exact-verified durable generation or recovery write; marker-time capture must match that digest and freezes the absolute path, strong file identity, and complete root-to-parent no-follow directory-identity chain in one immutable proof. A same-content replacement before marker capture is harmless and becomes the marker-time identity because it contains the exact source-authorized payload with the required owner and mode; a payload-changing replacement is rejected. That frozen proof is the sole authoritative publication snapshot. A compatibility path attribute may mirror `proof.artifact`, but readers never require or trust it, so a stale compatibility value cannot hide a valid proof and a bare legacy path never becomes current recovery evidence. A same-content inode replacement after capture or any ancestor edge replacement fails closed and clears the proof. Propagation, timeout snapshots, cleanup failures, and finalization publish, copy, replace, or clear the proof atomically.

A bounded heartbeat join timeout leaves the lock lease release-started rather
than release-complete. The owner performs one further bounded cleanup attempt
while preserving the first timeout as the primary diagnostic. If both joins
time out, classify cleanup as inconclusive. Intentionally retained shared
refresh-lock directories never authorize a lexical recovery or cleanup
pathname; report only descriptor-bound residue. Pause for controlled operator
cleanup only after confirming that no credential writer remains. Never
silently mark possibly orphaned directories as completed cleanup.

## Snapshot And Safety

- A catalogued local-login attempt owns one host refresh transaction before its first credential read. The same certified primary and legacy lock lease spans carrier selection, runtime credential exposure and network refresh, durable recovery staging, process-group or broker quiescence, the Linux/WSL2 watcher and final drain, and verified host commit. macOS persistence and snapshot checks and Linux/WSL2 writeback reuse the outer lease rather than nesting another host acquisition; staged-carrier locks remain independent. A second helper cannot copy or expose the host credential until the first transaction ends, and it must read the post-transaction host state after acquiring its own lease. A safe no-rotation attempt or a fully verified latest rotation releases normally. Unproven writer quiescence, an unpersisted rotation, retained carrier, or cleanup uncertainty abandons the lease: heartbeat and owned descriptors stop, the shared lock directories remain as a stale fence, and the lane pauses without automatic deletion, login guidance, or Copilot fallback. Every intentionally retained shared refresh lock reports descriptor-bound residue only; path-owned and borrowed-descriptor anchors alike provide no lexical recovery or cleanup pathname. API-key mode skips the local-login transaction.
- The helper requires `--base-ref` and `--head-ref`, resolves both to commits before launch, and rejects the range unless base is an ancestor of head. A diverged range reports the merge base to use instead of silently reviewing a two-endpoint diff with unrelated target-branch changes.
- It creates a `.git`-free frozen snapshot by streaming raw tree blobs from the head under source-repo `.codex-tmp/`; Git archive attributes, checkout filters, hooks, and repository config cannot rewrite that snapshot. Before writing each regular blob, it enforces a 64 MiB per-file limit plus a 512 MiB/100,000-entry total snapshot budget, and caps streamed tree metadata at 128 MiB. It also limits complete changed-path and raw-diff metadata to 128 MiB and 100,000 logical paths and streams a binary metadata-only `--submodule=short` artifact for the exact range through a 128 MiB output cap. Every head path must also fit the cleanup recursion capacity, including the extra materialized directory level for a gitlink, and an over-depth entry is rejected before that entry is materialized. Snapshot, tree, diff, and materialization budget exhaustion fails prelaunch and removes the partial container. Each secret discovery or count pass uses its existing 512 MiB blob-input budget as an admission-only limit; exhaustion makes secret admission `inconclusive` and must not remove an otherwise valid snapshot or delay, suppress, or redact reviewer input.
- `.codex-tmp` must be a real in-repository directory owned by the current user, never a symlink or group/other writable. Each random review container is created relative to a verified no-follow directory descriptor with owner-only `0700` permissions.
- The helper rejects a range when either base or head uses its reserved top-level `.codex-review` control path. Every head snapshot also rejects any path component beginning with the helper-owned `.codex-review-cleanup-` quarantine prefix so tracked content cannot collide with cleanup recovery state; a base-only path deleted by the reviewed range remains available in the trusted raw diff and the ephemeral complete-path proof until preflight scrubs it.
- The rendered review prompt is capped at 64 KiB before any reviewer starts. This keeps Copilot's required `--prompt` argument below per-argument execution limits and prevents an oversized override from reaching any external lane.
- Snapshot preparation runs Git with a cleaned environment, replacement objects disabled, disabled hooks/fsmonitor/external diff, and explicit `--no-ext-diff --no-textconv`. It does not checkout files through repository filters, keeps materialization failures in bounded typed diagnostics, and removes a partial review container on ordinary errors, forwarded signals, or `KeyboardInterrupt`. Completed snapshot ownership is handed to the caller while forwarded signals are blocked, before preparation returns. If removal fails before the state path is published, the terminal error names the retained container and cleanup failure instead of silently losing recovery evidence.
- The helper never initializes or fetches submodules. Gitlink changes publish only the path, mode, and base/head commit object IDs; the frozen diff never includes nested submodule logs, diffs, or content, even when the source checkout already has an initialized submodule.
- Every reviewer receives the same bounded findings-only prompt and primary diff file inside a helper-owned `.codex-review/` directory in the frozen snapshot. Custom prompt placeholders are expanded in one pass so placeholder-shaped text inside generated paths is never interpreted again.
- The frozen workspace contains no `.git` metadata. Codex receives only that workspace under a verified read-only filesystem profile; parent paths are not readable, writes and network are unavailable, and the supplied diff is the primary evidence surface. The runtime does not copy a Git shim or any other helper source into any reviewer-visible workspace. A Git command that cannot operate without repository metadata is expected lane context, not a reason to broaden permissions. Snapshot Git commands disable lazy fetching, terminal prompts, and askpass helpers, so a partial clone with missing objects fails closed instead of contacting a promisor remote or waiting for authentication.
- Every child receives a runtime-specific minimal environment allowlist instead of the parent's complete environment. Uppercase and lowercase proxy variables (with lowercase overrides taking precedence) plus the shared standard/custom CA bundle pointers are preserved so configured routing and TLS trust still work. Claude and Copilot authentication variables are never present in each other's process. Codex may receive `OPENAI_API_KEY` for headless authentication, but model-proposed shell commands cannot inherit it. `NODE_EXTRA_CA_CERTS` is a Claude-only exception rather than a shared reviewer input: Codex, Copilot, the pre-provenance Claude candidate probe, GPG, and security-sensitive host tools do not inherit it. `SSL_CERT_FILE` remains an independent caller input and never causes the Node variable to be set. This input/forwarding separation is not a claim that the final Linux sandbox maintains two trust domains: when Node extra CA is explicit, the validated material intentionally converges into the single private bundle described below. Arbitrary `NODE_*`, `NODE_TLS_REJECT_UNAUTHORIZED=0`, mTLS private-key inputs, and private-key material remain unsupported.
- Claude never mounts a caller-provided TLS path directly. A configured CA file, including `NODE_EXTRA_CA_CERTS`, is copied from a stable no-follow descriptor through the same absolute-path, owner/mode, size, PEM-only, private-key rejection, and atomic private-`0600` gates. A configured CA directory is enumerated through a fixed descriptor with bounded entries; bounded OpenSSL hash-link chains may cross into normal system or enterprise certificate locations, but every traversed link/directory and final regular file is owner/mode/identity checked around the read. Only certificate blocks are retained, and hash basenames become private `0600` regular files. On macOS, Seatbelt grants `file-read*` only to the exact helper-owned Node CA copy; it does not grant a parent-directory subpath read, and the caller path is absent from the profile and launch arguments. Literal ancestor metadata checks remain part of safe path traversal. Linux/WSL2 builds one helper-private bundle and captures plus revalidates its complete resolved path identity while serializing the read-only `bubblewrap` command. Node-only configuration retains default system trust; configured replacement inputs (`CURL_CA_BUNDLE`, `GIT_SSL_CAINFO`, `REQUESTS_CA_BUNDLE`, `SSL_CERT_FILE`, or `SSL_CERT_DIR`) preserve the existing replacement semantics before Node certificates are appended and deduplicated. The sandbox sets `NODE_EXTRA_CA_CERTS` only when the caller explicitly configured it, points it to `/etc/ssl/certs/ca-certificates.crt`, and adds no host mount. This freezes the resolved source consistently with runtime-library mounts; it is not an FD-bound atomic mount handoff against another same-euid process.
- Codex runs with a custom permission profile: only platform-minimal paths and the complete frozen tracked workspace are readable; `.git` remains denied, while tracked `.codex`, `.agents`, and environment files are intentionally readable as repository context; network and writes are unavailable. Because the frozen workspace is materialized from the exact head tree, this does not expose source-worktree untracked files, the host home, unrelated repositories, or reviewer/runtime credential sources. Codex ignores user config and execpolicy rules, sets `project_doc_max_bytes=0` so the reviewed head cannot inject `AGENTS.md` instructions, uses `approval_policy=never`, and gives model-proposed shell commands a fresh environment with an empty helper-owned `HOME`. The helper-owned review prompt is the only project-level review contract. The terminal artifact is rejected unless the persisted Codex turn context exactly matches the expected platform-minimal read, frozen-root read, `.git` deny, at most one direct `codex-arg0*` transport file under `$CODEX_HOME/tmp/arg0`, restricted network, and never-approve policy; any extra path or glob entry and any legacy or managed `sandbox_mode` override fail closed as `permission-mismatch`.
- Claude Code CLI compatibility is version-range and capability based. A release version `>=2.1.211,<3.0.0` is eligible only after the fixed Anthropic GPG fingerprint verifies its per-version signed manifest and the manifest checksum matches the installed platform binary. The helper does not pin `latest` or one patch release. Native Mach-O/ELF format, execute permission, identity output, architecture, and Linux/WSL2 libc compatibility reject scripts, interpreter wrappers, and incompatible artifacts; those checks never establish publisher provenance. macOS accepts an official thin arm64 or x64 artifact and selects the signed manifest entry from that artifact's architecture; x64 may run through Rosetta on Apple Silicon, so exact host-CPU equality is not required. Linux/WSL2 still requires a matching host architecture and libc. macOS code signing and notarization are not current acceptance gates. - The fixed-path GPG source is a native host-trust dependency, not Anthropic publisher provenance. Its resolved executable must be a root- or current-user-owned regular native file, executable and not group/world writable; the helper verifies stable file and path identities.   Every structurally valid broker `W` generation admitted to durable staging atomically reserves attempt-scoped durable-journal quota before any filesystem write, then commits to a distinct helper-private recovery carrier with synchronized file, directory, rename, and exact-readback state. On Darwin, every such sync runs `fsync` followed by required `F_FULLFSYNC`; failure NACKs before publication, while an already-renamed exact payload may be retained only as recovery-only evidence after acknowledged-path re-proof. The journal is bounded to eight generations and 8 MiB of payload; the last generation and 1 MiB are reserved for a terminal exact recovery copy, and a reservation is never refunded during the attempt. At the locked generation linearization point, only a still-current, non-abandoned ordinary generation is published and acknowledged. The current update that first enters the terminal reserve atomically closes later broker writes, commits into that reserve, and NACKs without publication; superseded contenders consume nothing and later requests are explicitly NACKed before callbacks. The terminal generation invalidates older staged host-writeback state and pauses inspection-inconclusive without fallback. The latest exact-verified generation is always the canonical current carrier. After server and handler quiescence, one owner removes non-authoritative complete entries only after proving that canonical carrier, then guarded-writes the latest acknowledged credential under the certified locks and removes the final recovery carrier only after verified host commit through the same full-sync boundary. If no staged host candidate remains, it reports the canonical carrier as the sole current recovery artifact; stale cleanup failure is separate, and a control-flow interruption that leaves any stale carrier unvisited promotes cleanup reporting to the recovery-root scope. Same-refresh-token carriers remain one logical login, supported Claude writers serialize through the audited lock window, and unrelated writers that bypass both locks remain outside the compare-before-write guarantee.

Unquiescent shutdown irreversibly closes the pending-update publication/ACK gate before boundedly draining handlers already inside the commit critical section. Any durable journal entry or quiescence stage makes the final abandonment error report that root, including beside an exact current carrier. Timeout-bearing helper calls must use this logged process-group path, every output-limit configuration must include a finite timeout, and the simpler captured-output path rejects timeout configuration before launch. Bounded-output arguments are validated before existing log files are opened. Prompt stdin uses an interruptible nonblocking writer, and output drains use interruptible nonblocking pipe reads; all I/O threads start inside a forwarded-signal mask and are registered only after successful startup, so a detached descendant cannot keep timeout cleanup blocked by retaining a pipe. Stateful status tails seek and decode only a bounded suffix instead of loading complete logs.
- Logged reviewer processes run in their own process groups. The helper forwards `SIGTERM`, `SIGINT`, `SIGHUP`, and `SIGQUIT`, reaps the reviewer, and terminates leftover descendants before restoring signal handlers; a stateful runner persists `128 + signal` as its terminal exit code. Foreground mode installs the same cancellation boundary before snapshot preparation and blocks signals across final cleanup, so preparation/preflight cancellation cannot strand a tracked-tree container.
- `stateful start` remains cancellable during workspace preparation. It defers a termination signal only across the narrow `Popen` handoff, checks it before publishing, and stops plus cleans any unpublished runner. The state becomes published only after the caller successfully flushes its path. The background runner keeps signal handling installed across preflight, executable discovery, model-attempt gaps, and child execution, then records `128 + signal` under a blocked finalization step.
- Claude/Copilot structured output is accepted only when no explicit `error`, `failed`, or `failure` state is present; partial text attached to an error result cannot become a clean final artifact.
- If Claude reports auxiliary model usage, effective-model verification selects the explicitly requested model entry rather than treating the first auxiliary entry as the reviewer model.
- Executable discovery validates `--version` identity and never trusts arbitrary repository `PATH` entries. It checks bounded system and user install locations. Codex and Copilot may use identity-checked native or interpreter-backed installations through a validated runtime path; Claude Code must resolve to a native Mach-O or ELF executable, so shebang scripts and interpreter wrappers are rejected before provenance checks. Explicit absolute overrides are `CODEX_REVIEW_CODEX_PATH`, `CODEX_REVIEW_CLAUDE_PATH`, and `CODEX_REVIEW_COPILOT_PATH`; an invalid explicit override or an explicit Claude override missing its GPG, probe-sandbox, or trusted review-tool prerequisite blocks the lane even when an automatically discovered absence could be runtime unavailability.
- Source files are never edited. The detached workspace is removed after `stateful wait` unless `--keep-workspace` is set or the legacy helper Codex runtime exits `127` after a matching successful preflight. That state is machine-visible in `stateful status`; it permits helper-specific diagnosis only and cannot satisfy or supply the canonical fresh-context Codex lane. Run `stateful cleanup` after diagnosis. A finite wait deadline covers runner completion, cleanup-lock acquisition, and workspace removal; `stateful final` applies its own bounded cleanup deadline. Timed cleanup runs in an independent worker that inherits the held cleanup-lock descriptor; lock ownership is handed off while forwarded signals are blocked, so a timeout or interrupted parent cannot let another waiter start a concurrent `rmtree`. Cleanup state-directory identity binds path and open descriptor with stable device, inode, mode, and owner metadata; entry churn beneath the same directory must not invalidate that binding, while path replacement must still fail closed. Cleanup-only legacy compatibility admits an empty owner-owned mode-`0664` `cleanup.lock` only through no-follow descriptors after proving a non-group/other-writable owner-owned `.codex-tmp` root and an exact-mode-`0700` state directory; after the exclusive lock is acquired, it revalidates both directories and the lock identity/mode before `fchmod(0600)`, `fsync`, and exact mode-`0600` validation. Every other group/other-writable or nonempty legacy lock fails closed without workspace removal. A daemon reaper collects the worker while long-lived callers continue. A successful worker clears any stale cleanup-error artifact while it still owns the cleanup lock.
- Logs, preflight evidence, attempts metadata, and `final.txt` remain in the state directory after workspace cleanup. Every newly created bound runtime artifact, control-state temporary, private slot, attempt stream, compatibility lock, and credential-update temporary is forced to exact mode `0600` on its held descriptor before publication or identity handoff, independent of a caller umask that masks owner permissions. A newly created `attempts` directory is likewise forced to `0700` inside the already-bound owner-private container before descriptor reopening, so an owner-masking umask cannot make the first launch inaccessible; a pre-existing name is opened and validated before any descriptor-based mode normalization. The stateful runner likewise forces both runner stdout/stderr descriptors to exact `0600` before spawning the child. Existing locks and artifacts are validated rather than chmod-repaired. Cleanup acquires the preparation-bound container directory lock before it creates, opens, or validates `cleanup.lock`, then acquires the compatibility lock; this fixed order serializes first-use initialization without weakening no-follow or path-identity checks, and timed worker handoff preserves the same descriptor order. Runner-liveness observers use shared nonblocking probes against the runner's exclusive lease, so concurrent `status` or `wait` probes cannot impersonate a live runner while a real runner still blocks cleanup. `attempts.json` is atomically rewritten after every completed model attempt through the same preparation-bound, exact-readback runtime-artifact writer used for the state marker, so cancellation during a later entitlement fallback preserves the earlier model/category/effective-runtime evidence without following a replaced container or parent path.

## Terminal States

### Reviewer Final

- exit `0`: a non-empty terminal final artifact exists
- exit `75`: transient/capacity failure; retry only the same runtime/model if the parent policy permits
- other nonzero: blocked or failed; inspect `stateful status`, `attempts.json`, and bounded logs
- `stateful final` prints only the saved terminal artifact on success
- exit `127` with `fallback_workspace_retained: true` means preflight succeeded but the legacy helper Codex executable was deterministically unavailable; inspect or clean that helper state, but never use the retained `.git`-free scope as the canonical Codex lane
- workspace cleanup failure is terminal nonzero even when the reviewer produced a clean artifact; the error and retained state directory remain visible for recovery

### Helper-State Secret Admission

- Run `stateful admission --state-dir <state_dir>` only as the second current-head check after harvesting the reviewer artifact with `stateful final`.
- exit `0`: the bounded public `secret_delta` summary is valid and `clean`, and the exact `preflight.json` bytes match the schema-v5 runner-sealed receipt; this is a successful optional helper-state check, not the required PR/master/merge-ready admission producer.
- exit `1`: the valid bounded summary reports one or more violations.
- exit `3`: the runner lock is still held, so the state is pending regardless of whether a preflight receipt has already been published; no terminal admission evidence exists until lock release.
- exit `75`: the valid bounded summary reports an inconclusive audit, the state is terminal but its preflight was not sealed, a schema-v4 marker lacks a receipt, or safe bounded admission evidence cannot otherwise be established.
- A successful reviewer final remains valid review evidence when admission exits `1` or `75`; neither result may be reclassified as reviewer failure. Foreground output is not admission evidence, and any head change invalidates both checks.

Each attempt records runtime, requested/effective model, requested/effective effort when observable, category, exit status, and log paths. Claude entitlement evidence comes only from a launched final-review invocation with strict structured-error and requested/effective-model verification. For Codex, the helper resolves the emitted thread ID to its persisted rollout and requires matching `turn_context` model and effort before accepting the final artifact. Missing verification on a successful result is `runtime-unverified`; any observed model, effort, or permission mismatch overrides even an otherwise entitlement-shaped failed attempt and stops the lane. None of those conditions is entitlement evidence, so none enters the fallback path.

Claude Code publisher provenance is anchored to Anthropic's release-signing primary-key fingerprint `31DD DE24 DDFA B679 F42D 7BD2 BAA9 29FF 1A7E CACE`. For the exact detected version, the helper verifies `manifest.json.sig` over `manifest.json` from `downloads.claude.ai`, then requires the installed platform binary to match the manifest checksum. Native format and platform compatibility checks reject script/interpreter wrappers and incompatible artifacts but cannot distinguish every native launcher. macOS platform signing and notarization are possible future defense-in-depth checks, not current acceptance gates, and never substitute for the signed manifest. No Claude candidate may receive credentials or tracked review content until provenance and bounded capability probes pass.

## Deliberate Omissions

The helper no longer supports generic `auto` lanes, OpenCode, Cursor Agent, `gh-copilot`, `codex-parallel`, live working-tree snapshots, reviewer-visible Git shims, arbitrary child argv, report sinks, or legacy helper names. These surfaces caused ambiguous review counting, model drift, or unnecessary runtime code exposure.
