# Isolated Review Helper Contract

The canonical helper is:

```bash
$HOME/.codex/skills/review-orchestration-playbook/scripts/isolated_review
```

It runs exactly one logical local reviewer against one frozen Git range. The parent skill composes multiple logical lanes.

## CLI

Foreground:

```bash
isolated_review \
  --repo /path/to/repo \
  --reviewer codex \
  --base-ref <base_sha> \
  --head-ref <head_sha>
```

Stateful:

```bash
isolated_review stateful start \
  --repo /path/to/repo \
  --reviewer claude \
  --egress-consent double-review \
  --base-ref <base_sha> \
  --head-ref <head_sha>

isolated_review stateful status --state-dir <state_dir>
isolated_review stateful wait --state-dir <state_dir>
isolated_review stateful final --state-dir <state_dir>
isolated_review stateful cleanup --state-dir <state_dir>
```

Synthetic-token catalog inspection is read-only:

```bash
isolated_review synthetic-tokens validate
isolated_review synthetic-tokens list --json
isolated_review synthetic-tokens get <id> --json
isolated_review synthetic-tokens list-exemptions --json
isolated_review synthetic-tokens audit-master --repo <path> --ref <sha> --exemption <id>
```

Review commands accept repeatable `--synthetic-secret-exemption <id>` selections for helper-owned historical migration envelopes. Read [synthetic-token-fixtures.md](synthetic-token-fixtures.md) before changing the catalog or selecting an exemption.

Always pass `--state-dir`; it is not positional. `stateful wait --timeout-seconds` accepts only a non-negative finite value, bounds the caller's wait, and does not kill or downgrade a healthy reviewer. `stateful cleanup` explicitly removes a workspace retained by `--keep-workspace` or for a clean-context fallback while preserving state artifacts.

The parent acquires an exclusive runner lock before spawn and passes its file descriptor to the child for the child's full lifetime. Cross-process `status` / `wait` trusts that lock, not PID existence, so a reused PID cannot masquerade as the review runner.

## Logical Reviewers

`--reviewer codex`:

1. `gpt-5.6-sol`, `xhigh`
2. `gpt-5.5`, `xhigh`, only after explicit model entitlement/policy denial

`--reviewer claude`:

1. Claude Code `claude-opus-4-8`, `max`
2. Claude Code `claude-opus-4-7`, `max`, only after explicit account, plan, organization-policy, or model-entitlement denial for Opus 4.8
3. Copilot CLI `claude-opus-4.8`, `max`, only when Claude Code is absent, has no usable local/API authentication, or both Claude Code Opus models are entitlement-blocked
4. Copilot CLI `claude-opus-4.7`, `max`, only after the same explicit account, plan, organization-policy, or model-entitlement denial for Opus 4.8

Claude Code CLI releases are not pinned to one current patch. The Claude runtime runs in safe mode and accepts publisher-verified releases `>=2.1.187,<3.0.0` only when the detected platform, outer sandbox, credentials, launch flags, safe-mode semantics, and structured output contract pass. Codex CLI and Copilot CLI likewise have no exact executable-version pin; their existing identity, capability, and output verification remains authoritative. These CLI policies do not change the explicit reviewer-model policy above: model IDs remain intentionally pinned.

The Copilot IDs follow GitHub's authoritative [supported models matrix](https://docs.github.com/en/copilot/reference/ai-models/supported-models), which lists Opus 4.8 and Opus 4.7 for Copilot CLI. The shorter command-reference example table can lag product availability and is not treated as a runtime allowlist; exact model verification still fails closed if the installed CLI rejects or substitutes any ID.

The Claude-family lane requires one of `--egress-consent explicit-claude-review`, `--egress-consent double-review`, or `--egress-consent triple-review`. The helper saves this value in state and writes `egress.json`; it refuses to start the external lane without it. `explicit-claude-review` authorizes Anthropic only, while `double-review` and `triple-review` also authorize GitHub Copilot fallback when Claude Code is unavailable, has no usable local/API authentication, or both pinned Claude Opus models are entitlement-blocked.

Model verification normalizes punctuation only and then requires exact equality. A requested `gpt-5.5` never accepts `gpt-5.5-mini`, `gpt-5.5-codex`, or any other suffix as the same model.

Capacity, overload, rate limits, timeouts, network/5xx errors, missing artifacts, silent model substitution, and review findings never trigger a model downgrade. The helper records every attempt and reports transient failures without switching models.

Fallback classification uses stderr plus explicit structured CLI error events and error-schema fields only. Partial `result`/`data` payloads, reviewer tool output, and repository text on stdout are never scanned for entitlement or transient substrings.

For Claude runtime preparation, automatic absence, unsupported platform enforcement, a cleanly missing capability from an automatically discovered candidate, or unusable authentication may be eligible for the already-authorized Copilot fallback. Publisher fingerprint/signature/checksum failures, contradictory safe-mode claims, unsafe credential metadata, outer-sandbox mismatches, and invalid explicit overrides are blocked security or configuration failures and never authorize fallback. An explicit `CODEX_REVIEW_CLAUDE_PATH` that lacks the required GPG verifier, outer probe sandbox, or trusted review-tool prerequisite is also blocked rather than reclassified as automatic absence. Probe, manifest-download, executable resolve/stat I/O, file-race, and bounded-I/O failures are inconclusive and likewise never authorize fallback.

A current-attempt authentication warmup that is transient, times out, exceeds an output limit, fails to drain, or leaks a process is inconclusive and returns exit `75`; it never authorizes Copilot fallback. The same preflight-inconclusive rule applies when either the initial or post-warmup credential freshness read times out, exceeds an output limit, fails to drain, or leaks a process. If an earlier Claude model already completed with entitlement metadata, that attempt remains in `attempts.json` while the unstarted model is not fabricated as an attempt. A fixed-input warmup that instead returns an explicit entitlement or organization-policy denial may become model-chain fallback evidence only when its strict top-level result is explicitly in an error state, the structured error fields classify as entitlement without relying on stderr, and model metadata identifies the exact requested effective model. A successful result plus entitlement-shaped stderr is not fallback evidence. The helper records verified evidence as an `entitlement` attempt with no final text and without claiming that the final broker or repository-review sandbox launched. The warmup's 64 KiB per-stream output cap is below the complete in-memory capture limit, so its full stdout/stderr is retained before temporary cleanup and then written to the formal attempt log paths. Missing or mismatched model metadata becomes `runtime-unverified` or `model-mismatch`, stops the lane, and never authorizes fallback. An explicit authentication failure remains unavailable even if the refresh produced a structurally fresh Keychain item. Every later model still repeats its own credential-boundary refresh and validation. An explicitly unavailable current-attempt credential may use the existing Copilot fallback only with `double-review` or `triple-review` consent; `explicit-claude-review` remains Anthropic-only.

An attempt-local restricted Keychain broker failure has the same unavailable and consent-gated fallback semantics as an unavailable credential. It writes `authentication-preflight-unavailable`, preserves earlier attempt evidence, and never becomes an inconclusive model attempt when the model was not launched.

A structured transient warmup remains inconclusive even if the post-warmup credential read also finds that the restricted broker disappeared; transient evidence never authorizes fallback. A successful or otherwise non-entitlement, non-transient warmup preserves the explicit broker-unavailable result. During final runtime staging, credential-read timeout, output-limit, drain, or process-leak failures write `authentication-preflight-inconclusive` with no fabricated model attempt. Restricted-broker or loopback failure before the Claude CLI launch writes `authentication-preflight-unavailable` and follows the existing consent gate.

## Snapshot And Safety

- The helper requires `--base-ref` and `--head-ref`, resolves both to commits before launch, and rejects the range unless base is an ancestor of head. A diverged range reports the merge base to use instead of silently reviewing a two-endpoint diff with unrelated target-branch changes.
- It creates a `.git`-free frozen snapshot by streaming raw tree blobs from the head under source-repo `.codex-tmp/`; Git archive attributes, checkout filters, hooks, and repository config cannot rewrite that snapshot. Before writing each regular blob, it enforces a 64 MiB per-file limit plus a 512 MiB/100,000-entry total snapshot budget, and caps streamed tree metadata at 128 MiB. It also limits changed-path and raw-diff metadata to 128 MiB and 100,000 changes, streams a binary `--submodule=diff` artifact for the exact range through a 128 MiB output cap, and bounds changed-blob secret scanning to 512 MiB total. Exceeding any budget fails closed and removes the partial container.
- `.codex-tmp` must be a real in-repository directory owned by the current user, never a symlink or group/other writable. Each random review container is created relative to a verified no-follow directory descriptor with owner-only `0700` permissions.
- The helper rejects a range when either base or head uses its reserved top-level `.codex-review` control path.
- The rendered review prompt is capped at 64 KiB before any reviewer starts. This keeps Copilot's required `--prompt` argument below per-argument execution limits and prevents an oversized override from reaching any external lane.
- Snapshot preparation runs Git with a cleaned environment, replacement objects disabled, disabled hooks/fsmonitor/external diff, and explicit `--no-ext-diff --no-textconv`. It does not checkout files through repository filters, redacts secret-shaped paths from materialization diagnostics, and removes a partial review container on ordinary errors, forwarded signals, or `KeyboardInterrupt`. Completed snapshot ownership is handed to the caller while forwarded signals are blocked, before preparation returns. If removal fails before the state path is published, the terminal error names the retained container and cleanup failure instead of silently losing recovery evidence.
- Submodules remain uninitialized and unfetched; their gitlink changes are represented in the frozen diff.
- Every reviewer receives the same bounded findings-only prompt and primary diff file inside a helper-owned `.codex-review/` directory in the frozen snapshot. Custom prompt placeholders are expanded in one pass so placeholder-shaped text inside generated paths is never interpreted again.
- The frozen workspace contains no `.git` metadata. Codex receives only that workspace under a verified read-only filesystem profile; parent paths are not readable, writes and network are unavailable, and the supplied diff is the primary evidence surface. The runtime does not copy a Git shim or any other helper source into any reviewer-visible workspace. A Git command that cannot operate without repository metadata is expected lane context, not a reason to broaden permissions. Snapshot Git commands disable lazy fetching, terminal prompts, and askpass helpers, so a partial clone with missing objects fails closed instead of contacting a promisor remote or waiting for authentication.
- Every child receives a runtime-specific minimal environment allowlist instead of the parent's complete environment. Uppercase and lowercase proxy variables (with lowercase overrides taking precedence) plus standard/custom CA bundle pointers are preserved so configured routing and TLS trust still work. Claude and Copilot authentication variables are never present in each other's process. Codex may receive `OPENAI_API_KEY` for headless authentication, but model-proposed shell commands cannot inherit it.
- Claude never mounts a caller-provided TLS path directly. A configured CA file is copied from a stable no-follow descriptor. A configured CA directory is enumerated through a fixed descriptor with bounded entries; bounded OpenSSL hash-link chains may cross into normal system or enterprise certificate locations, but every traversed link/directory and final regular file is owner/mode/identity checked around the read. Only certificate blocks are retained, hash basenames become private `0600` regular files, and private-key markers are rejected. Linux/WSL2 builds one helper-private bundle and captures plus revalidates its complete resolved path identity while serializing the read-only `bubblewrap` command. This freezes the resolved source consistently with runtime-library mounts; it is not an FD-bound atomic mount handoff against another same-euid process.
- Codex runs with a custom permission profile: only platform-minimal paths and the frozen workspace are readable; `.git`, `.codex`, `.agents`, and environment-file globs are denied; network and writes are unavailable. It ignores user config and execpolicy rules, sets `project_doc_max_bytes=0` so the reviewed head cannot inject `AGENTS.md` instructions, uses `approval_policy=never`, and gives model-proposed shell commands a fresh environment with an empty helper-owned `HOME`. The helper-owned review prompt is the only project-level review contract. The terminal artifact is rejected unless the persisted Codex turn context exactly matches the expected restricted filesystem entries, at most one direct `codex-arg0*` transport file under `$CODEX_HOME/tmp/arg0`, denied control paths, restricted network, and never-approve policy; extra read paths/globs and legacy or managed `sandbox_mode` overrides fail closed as `permission-mismatch`.
- Claude Code CLI compatibility is version-range and capability based. A release version `>=2.1.187,<3.0.0` is eligible only after the fixed Anthropic GPG fingerprint verifies its per-version signed manifest and the manifest checksum matches the installed platform binary. The helper does not pin `latest` or one patch release. Native Mach-O/ELF format, execute permission, identity output, architecture, and Linux/WSL2 libc compatibility reject scripts, interpreter wrappers, and incompatible artifacts; those checks never establish publisher provenance. macOS accepts an official thin arm64 or x64 artifact and selects the signed manifest entry from that artifact's architecture; x64 may run through Rosetta on Apple Silicon, so exact host-CPU equality is not required. Linux/WSL2 still requires a matching host architecture and libc. macOS code signing and notarization are not current acceptance gates. They may be added later as optional defense in depth, but cannot replace the signed manifest. Read [claude-runtime-trust.md](claude-runtime-trust.md) for the complete ordering and trust anchors.
- The fixed-path GPG source is a native host-trust dependency, not Anthropic publisher provenance. Its resolved executable must be a root- or current-user-owned regular native file, executable and not group/world writable; the helper verifies stable file and path identities. Parent directories must be root/current-user owned and non-group/world writable, except that canonical Homebrew parents on macOS may be group-writable only by the exact `admin` group, whose members are explicitly part of the host-tool TCB. After inspection, the helper keeps a stable source descriptor, copies the main executable from that descriptor into a fresh current-user `0700` GPG home directly below an explicit helper-owned `0700` root in the isolated review state, atomically publishes a mode-`0500` execution snapshot, and uses only that snapshot for key dearmoring, fingerprint listing, and detached-signature verification. The main snapshot fixes the executable file, not its loader dependencies. On macOS, fixed root-owned `/usr/bin/otool` recursively collects a bounded dependency closure; the main executable must contain exactly one `LC_LOAD_DYLINKER` naming `/usr/lib/dyld`, dependencies must not contain that command, and relative dyld names, rpaths, dyld environment load commands, and paths outside sealed `/usr/lib`/`/System/Library` or canonical Homebrew `opt`/`Cellar` roots are rejected. Every non-sealed lexical/resolved path identity is revalidated before each GPG call. A safe `otool` that cannot start or returns nonzero is inspection-inconclusive, not clean runtime unavailability. On Linux/WSL2, automatic selection is limited to root-owned `/usr/bin/gpg{,2}`. Stable-descriptor ELF inspection first proves that the loader-visible `PT_DYNAMIC` address, memory, and file-backed ranges map consistently through exactly one `PT_LOAD` at byte and loader-page granularity, including page-offset congruence and rejection of every other page-rounded mapping over the dynamic table; only then does it reject `DT_RPATH`/`DT_RUNPATH` and `DT_AUDIT`/`DT_DEPAUDIT` in the snapshot before any loader process starts. Host GPG must name the architecture's canonical glibc interpreter. The helper captures and proves that loader's lexical/resolved root-owned identity, Linux-native mount provenance, host architecture, ELF64 `ET_DYN` type, absence of a second interpreter or unsafe dynamic tags, and floating glibc version `>=2.27,<3.0`; musl and unknown host-GPG loaders are unavailable. It then invokes only `loader --list <snapshot>`, never an implementation-variable `ldd` script, captures every reported lexical symlink chain and resolved root-owned endpoint, rejects unsafe dependency tags/types and any interpreter other than that same canonical glibc loader after the proven list trace but before GPG, revalidates the old closure, and rebuilds and requires exact closure and loader-version equality before each GPG call. Only the private snapshot path may traverse a root-owned exact-`01777` system-temp ancestor; its non-final directory identities retain device/inode/type/mode/owner/group anchors while ignoring expected entry-count/time churn as GPG creates keyrings and locks. The final snapshot executable, glibc loader, and dependency chains keep complete metadata identities and remain non-writable. Ambient temporary-directory variables do not select the private root. WSL2 batch-validates both old and refreshed snapshot, loader, lexical dependency, and resolved endpoint paths against bounded mountinfo; Windows-backed provenance is invalid, while unavailable or malformed mount evidence is inconclusive. `claude-runtime.json` still records the inspected source path as `gpg_verifier`, with `gpg_verifier_trust: fixed-path-native-host-tool`, separately from `publisher_provenance: anthropic-signed-manifest`.
- Every untrusted identity and version probe runs before provenance in a helper-owned bootstrap sandbox with no credentials, frozen review workspace, host configuration, writes, or network. macOS uses a deny-by-default Seatbelt profile. Linux and WSL2 use a synthetic-root `bubblewrap` sandbox that exposes the candidate plus only narrowly selected root-owned, non-group/world-writable system library roots; it does not run `ldd` on the unverified binary. After the signed checksum passes, trusted root-owned `ldd` may collect the exact root-owned, non-group/world-writable dynamic dependencies for read-only mounting in the final Linux sandbox. Every collected library and each path component is identity-captured, ownership/mode checked, and revalidated before command construction and final mounting. WSL1 and native Windows are unsupported. Probe homes and output files are helper-owned, and time plus both output streams are bounded. A timeout, output-limit breach, drain failure, file race, or leaked descendant is inconclusive and cannot authorize Copilot fallback.
- GPG and Linux security-sensitive host tools use explicit minimal environments rather than inheriting the caller's process environment. The allowlist sets only fixed home, locale, path, and temporary-directory values needed by each tool, so dynamic-loader variables, shell startup configuration, compiler flags, and toolchain overrides cannot change GPG, glibc-loader probes, post-provenance `ldd`, `bubblewrap`, `socat`, `rg`, compiler probes, or launcher compilation.
- Immediately after the signed manifest, size, and SHA-256 checks pass, the helper copies the source candidate into a digest-keyed executable file inside a current-user-owned `0700` snapshot directory. The copy is rehashed against the signed artifact, published atomically with mode `0500`, and revalidated before reuse. The original candidate is never executed again: the same helper-owned verified snapshot is captured before the Claude model chain and runs bounded `--help`, post-provenance dependency discovery, authentication-bearing preparation, and every final model attempt. This closes both the source-path replacement window and any attempt-to-attempt rediscovery window.
- Capability validation accepts only the supported release range, requires every public flag used by the helper's real Claude commands to appear uniquely in bounded `--help` output, requires the permission-mode help to advertise `dontAsk`, and parses the required safe-mode semantics without exact-matching the complete help string. Safe mode must unambiguously disable user/project instructions, skills, plugins, hooks, MCP servers, custom commands and agents, and other local customizations while preserving authentication, model selection, built-in tools, and permissions. The customization sentence must match one bounded whole-sentence positive grammar with a known-item enumeration and a terminal disabled state; temporal reversals, extra predicates, unknown enumeration clauses, negation, duplication, contradictions, or weakening block the lane. Every sentence in the safe-mode option block is default-denied unless it independently matches one bounded grammar for customization disablement, managed-policy preservation, runtime preservation, the exact environment assignment, a positive safe-mode self/anaphoric claim, or a bounded documentation/information topic that remains available online. Harmless wrapping and the official bounded troubleshooting-purpose suffix remain accepted. Missing required behavior may make an automatic candidate unavailable; unsafe semantics never do. There is no credential-free fixed-input behavioral canary: the final real invocation and strict structured-output, effective-model, error-state, and terminal-artifact validation provide the behavioral acceptance check. macOS retains the hardening-compatible `default` permission mode and `Read,Grep,Glob`. Linux and WSL2 use `dontAsk`, expose only `Read`, explicitly allow only cwd-relative `Read(./**)`, reject ASCII `@` file mentions in the exact prompt bytes before authentication and again before each attempt, and deny `Grep`, `Glob`, and every non-workspace synthetic-root mount. Immediately before each sandbox command is serialized, they repeat a bounded no-follow workspace symlink audit: stable relative links that remain inside the frozen workspace are supported, while absolute links, `/proc` magic links, transiently escaping `..` paths, chain escapes, loops, races, and inspection failures fail closed. All platforms carry explicit model and effort arguments, disable hooks and setting sources, and omit browser, slash-command, inherited MCP, and session-persistence surfaces.
- The helper uses layered enforcement. The helper-owned outer sandbox exposes only the frozen workspace, helper-owned home/temp/control paths, required runtime files, the restricted review-tool path, validated certificate copies, and the platform credential carrier; it enforces host filesystem, process, write, and network isolation. The helper-controlled proxy permits the pinned authentication targets during the macOS-only no-tools warmup when it is needed, and only `api.anthropic.com:443` during the final review. On Linux and WSL2, the publisher-verified Claude permission engine is an additional trusted boundary because the same Claude process must read its credential carrier while model-visible file tools must not. Command construction requires exact `dontAsk`/tool/allow/deny settings, double-slash absolute root and recursive denies for every non-workspace mount, and no separate mount below `/workspace`; any future uncovered mount fails closed.
- Local-login credential freshness is an attempt-boundary property, not a reservation for the remaining model chain. The current model attempt must begin with at least its complete 30-minute timeout plus the 2-minute safety margin, currently 1920 seconds. On macOS, after verified executable, restricted-tool, and TLS preparation, every model attempt first checks the current-account Keychain item. A sufficiently fresh credential proceeds without refresh. Otherwise the helper runs the same fixed-input, no-tools, no-workspace-read safe-mode warmup with the current attempt's model, then re-reads and validates the Keychain item. The final capability-authenticated read-only broker re-reads and fail-closed validates the same single-attempt window before serving the credential once; it cannot expose Keychain services or refresh/update commands to the final process. Every later Opus attempt repeats this refresh-if-needed and validation sequence. On Linux and WSL2, every model attempt independently accepts only the current user's non-symlink regular credential file with exact mode `0600`, validates the same 1920-second window, copies it into a new helper-owned `0700` config directory as a private `0600` file, and mounts only that staged config read-only at `/config`; the host credential itself is never mounted. The file-tool policy denies `/config` to the model. An explicitly exported `ANTHROPIC_API_KEY` skips local-login warmup and staging; `/proc` and `/dev` are denied so the model cannot read the key through process environment or descriptor aliases. Linux and WSL2 perform no pre-chain authentication staging, warmup, refresh, or host-credential write. Positive WSL2 classification requires an explicit `wsl2` or `microsoft-standard` marker in the kernel release/version. `/run/WSL`, any nonempty raw `WSL_INTEROP` value (including an invalid or missing endpoint), distro environment, binfmt evidence, and a generic `microsoft` kernel identity are conservative weak WSL-presence signals. They do not authorize WSL2 and may classify ambiguous or spoofed state as unsupported WSL1. A guest that removes both the marker and every weak signal is observationally indistinguishable from native Linux and follows that classification. The discovered Claude executable, credential source, frozen review/state container, private GPG temporary root, and every helper runtime path—including the verified snapshot—must live in the WSL Linux filesystem. The common Linux/WSL guard rejects positive DrvFS/Windows provenance from bounded `/proc/self/mountinfo`, even when a markerless guest was classified as Linux. Positively identified WSL2 also keeps the fast literal `/mnt/<drive>` rejection and requires every path to prove local native Linux backing. Known DrvFS/Windows provenance, including custom automount, bind-mount, and alias paths, is blocked. Only WSL's local ext4 `/dev/sdX` storage and backing-free tmpfs are currently proven by the strict WSL2 inspection; overlay, FUSE, 9p, virtiofs, loop/device-mapper/network-backed, and unknown filesystems are inspection-inconclusive rather than guessed safe. Unreadable, malformed, oversized, non-canonical, or non-covering mount information also fails closed. A read-only mount prevents mutation, not disclosure to the trusted Claude process; stronger credential isolation would require an external authentication injector.
- Mountinfo mount points and ordinary filesystem roots must be absolute
  canonical paths. Exact `nsfs` records may instead carry one strictly bounded
  kernel namespace handle such as `net:[4026531840]`; this unused metadata field
  never participates in path coverage or Windows-provenance decisions. Opaque
  roots for every other filesystem and malformed namespace handles fail closed.
- Linux helper runtime directories follow create-or-validate semantics. A missing path is created once; an existing path must already be a real current-user directory with exact mode `0700` where private or otherwise no group/world write, and must retain the same no-follow descriptor/path identity across validation. The helper never follows a symlink or uses `chmod` to repair an attacker-controlled pre-existing path.
- The fixed no-shell Linux launcher blocks `SIGTERM`, `SIGINT`, `SIGHUP`, and `SIGQUIT` across proxy/workload fork and process-group publication, inspects pending signals before each handoff, restores default child signal state, forwards cancellation to both process groups, and terminates/reaps leftovers. This closes the signal-arrival windows that could otherwise strand the relay or review workload.
- `claude-runtime.json` distinguishes source discovery from accepted execution. It records `source_executable`, the helper-owned `verified_executable` snapshot, manifest/signature/checksum and the fixed-path GPG source as `gpg_verifier`. macOS writes `authentication-preflight-complete` only after the current attempt's API-key configuration or local-login refresh/validation, with `authentication.status` set to `configured` or `freshness-verified`, `authentication.model` naming the current model, and `validated_for_model` naming the model whose window passed; this overwrites any earlier entitlement model before final broker/profile preparation writes `runtime-launching` and `sandbox-auth-staged`. Publishing a later authentication-boundary outcome clears the previous attempt subtree and returns the outer-sandbox status to pending. A warmup supervision or transient classification writes `authentication-preflight-inconclusive` with the current model and stable warmup failure class without inventing a formal review attempt. A strictly verified warmup entitlement writes `authentication-preflight-entitlement`, keeps the outer sandbox pending, leaves `validated_for_model` unset because credential freshness did not pass, and records only the requested/effective model plus the entitlement attempt result with no final text. An explicitly unavailable macOS Keychain credential writes `authentication-preflight-unavailable`. Linux/WSL2 writes `runtime-ready` and `sandbox-auth-staged` only after the current attempt's private credential staging and isolation probe; credential unavailability prevents that phase and remains in the ordinary unavailable error artifact. A bounded supervisor timeout, output overflow, drain failure, or retained descendant during a launched Claude attempt writes `attempt-inconclusive` with a stable failure class. `attempt-complete` records final category/return code and requested/effective model and effort. No phase represents whole-chain credential freshness.
- Copilot runs with a fresh helper-owned `COPILOT_HOME`, so persisted permission grants and configuration cannot broaden the lane. It uses plan mode with `-C` pinned to the frozen workspace, a model-visible tool allowlist limited to `view`, `glob`, and `grep`, no `--add-dir` or `--allow-all-paths`, explicit shell/write/URL denial, temp-directory denial, and disabled custom instructions, built-in MCPs, bash environment loading, experimental features, and remote session export. Excluding the `skill` and delegation tools prevents project skills or agents from injecting reviewed-head instructions. Before every attempt, the helper runs a logged `copilot help permissions` probe with a 20-second deadline and 64 KiB per-stream cap, then confirms that availability filters control which tools the model can see, default file access is cwd-only, `--disallow-temp-dir` removes the implicit temp root, and deny rules override `--allow-all-tools`; otherwise the fallback fails closed. Secret-like auth variables are withheld from its tools.
- Claude output is accepted only when the entire output is the single documented top-level `result/success` RFC JSON object with `is_error=false`, non-empty `result`, no non-empty or malformed `error` / `errors` / `api_error_status` field, and non-empty model usage metadata whose keys are model IDs and whose values are usage objects; only JSON's ASCII whitespace is valid outside the object, and duplicate object keys plus non-standard constants such as `NaN` or `Infinity` are rejected recursively. Structured Claude or Copilot errors never supply final text, but their trustworthy top-level model metadata is retained so silent substitution still blocks fallback; an entitlement result without required model/effort verification becomes `runtime-unverified` and cannot authorize fallback. Copilot JSONL is split only on LF, treats only JSON's ASCII whitespace as blank, applies the same strict JSON rules, and requires every other record to be a JSON object. A complete-stream turn state machine rejects nested, interleaved, unmatched, or unclosed `assistant.turn_start` / `assistant.turn_end` boundaries before selecting the last completed turn; a structured error must belong to the current or final turn, and any completed `assistant.turn_end` must be the final JSONL record. It validates the last top-level `assistant.message` rather than skipping malformed later messages, and accepts it only when `toolRequests` is absent or empty. Every top-level session, message, and usage model value across the stream must agree; each new `session.start` resets session-scoped fallback evidence, so a session without `selectedModel` cannot inherit an earlier session's model. Missing model fields may use only documented evidence from the current session/turn, while any present non-string or empty `selectedModel` / `model` field makes both success and error output unverifiable. When no usage is emitted, verification uses the current CLI's message model or current `session.start.selectedModel` extension. Non-JSON prefixes or suffixes, arbitrary progress, tool, nested `message`, `content`, or `data` fields never become a terminal artifact. Parsed result content is preserved verbatim except for trailing CR/LF normalization in the persisted file. Any structured error clears the result text.
- Before any network-backed Codex, Claude Code, or Copilot run, the helper rejects any absolute, transiently escaping, or ultimately escaping symlink in the frozen workspace and blocks credential-like paths or high-confidence secret patterns found across base-to-head changed paths, both sides of changed raw blobs, every head-snapshot path name, symlink target, and file, the frozen diff, or prompt. Changed paths and raw-blob findings stream to NUL-delimited control files and are scanned incrementally, including deleted binary credentials and credential filenames nested under fixtures or copied home directories. The helper creates all six reviewer-visible `.codex-review` control files as owner-only `0600` independently of a permissive caller umask, requires that directory to contain exactly those files, binds the directory identity, mode, stable metadata, entry-set digest, and every file's name/size/digest/record evidence to raw-free helper-private state, then consumes each file through one no-follow, nonblocking, owner-matching regular-file descriptor with bounded reads and stable before/after metadata. Extra regular files, nested directories, symlinks, hardlinks, FIFOs, tampering, and type drift therefore fail before egress; prompt overrides use the same secure reader before rendering. Sensitive-content findings plus symlink materialization/escape diagnostics secret-check paths and targets before display; secret-bearing values use typed redaction markers, while control characters and undecodable bytes are escaped. The helper-owned finite synthetic-token catalog may suppress only an exact complete `generic-secret-assignment` capture for an authoring entry; every other scanner rule and credential-like path remains blocking. An unquoted accepted value must end at an unambiguous line boundary or end of input; immediate quote/backslash/backtick/expansion bytes, inline `#`/`;`, deeper-indented YAML or INI content, operator continuation, tabs, excessive trivia, and any next content other than a same-or-shallower named assignment, mapping key, document marker, or diff boundary fail closed under bounded inspection. An explicitly selected legacy envelope may suppress only catalog-declared `generic-secret-assignment`, `github-token`, or `jwt` captures after complete-tree counting proves both each raw head count and each count not embedded inside a longer value from the same envelope do not exceed their base counts. Equal global content counts may move across safe paths; copies, net additions, and extracting a registered substring from a longer value all block. Independently, every complete-catalog legacy raw value and canonical Base64 storage encoding is forbidden as a byte substring of any base, head, or materialized repository path, using a finite linear matcher before path decoding or display. Cross-envelope value overlap fails closed, and legacy matching never applies to the prompt. The stateful runner reloads helper-private count evidence and recomputes both count classes over the materialized head before egress. Findings and bounded acceptance evidence record only catalog versions, stable IDs, sides or surfaces, rules, SHA-256 digests, and occurrence counts, never raw matched values; raw-free checks cover every authoring value and every selected or unselected legacy raw/storage value in the complete catalog. A successful check writes retained `preflight.json` before executable discovery or model launch so a parent PR-readiness workflow can prove that preflight preceded its separate independent Codex lane.
- Reviewer stdout/stderr stream directly to complete per-attempt files. Only a bounded head/tail capture is retained in memory for generic error classification; the middle is never buffered in the runner. Every final Codex, Claude, and Copilot review launch has a 30-minute supervisor deadline and a 64 MiB limit on each output stream; timeout, overflow, drain failure, or retained descendants terminate the process group and are reported as inconclusive rather than leaving the stateful runner lock held indefinitely. The fixed-input Claude authentication warmup has its own bounded supervisor and becomes model-chain attempt evidence only for an exact-model-verified entitlement denial. The failed runtime/model plus its log paths are persisted before returning the inconclusive state. Copilot's complete JSONL is validated directly from that file with a one-pass, constant-state parser and a 4 MiB per-record cap, so aggregate output larger than the memory capture does not corrupt terminal/model verification. Timeout-bearing helper calls must use this logged process-group path, every output-limit configuration must include a finite timeout, and the simpler captured-output path rejects timeout configuration before launch. Bounded-output arguments are validated before existing log files are opened. Prompt stdin uses an interruptible nonblocking writer, and output drains use interruptible nonblocking pipe reads; all I/O threads start inside a forwarded-signal mask and are registered only after successful startup, so a detached descendant cannot keep timeout cleanup blocked by retaining a pipe. Stateful status tails seek and decode only a bounded suffix instead of loading complete logs.
- Logged reviewer processes run in their own process groups. The helper forwards `SIGTERM`, `SIGINT`, `SIGHUP`, and `SIGQUIT`, reaps the reviewer, and terminates leftover descendants before restoring signal handlers; a stateful runner persists `128 + signal` as its terminal exit code. Foreground mode installs the same cancellation boundary before snapshot preparation and blocks signals across final cleanup, so preparation/preflight cancellation cannot strand a tracked-tree container.
- `stateful start` remains cancellable during workspace preparation. It defers a termination signal only across the narrow `Popen` handoff, checks it before publishing, and stops plus cleans any unpublished runner. The state becomes published only after the caller successfully flushes its path. The background runner keeps signal handling installed across preflight, executable discovery, model-attempt gaps, and child execution, then records `128 + signal` under a blocked finalization step.
- Claude/Copilot structured output is accepted only when no explicit `error`, `failed`, or `failure` state is present; partial text attached to an error result cannot become a clean final artifact.
- If Claude reports auxiliary model usage, effective-model verification selects the explicitly requested model entry rather than treating the first auxiliary entry as the reviewer model.
- Executable discovery validates `--version` identity and never trusts arbitrary repository `PATH` entries. It checks bounded system and user install locations. Codex and Copilot may use identity-checked native or interpreter-backed installations through a validated runtime path; Claude Code must resolve to a native Mach-O or ELF executable, so shebang scripts and interpreter wrappers are rejected before provenance checks. Explicit absolute overrides are `CODEX_REVIEW_CODEX_PATH`, `CODEX_REVIEW_CLAUDE_PATH`, and `CODEX_REVIEW_COPILOT_PATH`; an invalid explicit override or an explicit Claude override missing its GPG, probe-sandbox, or trusted review-tool prerequisite blocks the lane even when an automatically discovered absence could be runtime unavailability.
- Source files are never edited. The detached workspace is removed after `stateful wait` unless `--keep-workspace` is set or the Codex runtime exits `127` after a matching successful preflight. That deterministic-unavailability state retains the immutable frozen workspace for the clean-context fallback and is machine-visible in `stateful status`; run `stateful cleanup` after the fallback decision and artifact are complete. A finite wait deadline covers runner completion, cleanup-lock acquisition, and workspace removal; `stateful final` applies its own bounded cleanup deadline. Timed cleanup runs in an independent worker that inherits the held cleanup-lock descriptor; lock ownership is handed off while forwarded signals are blocked, so a timeout or interrupted parent cannot let another waiter start a concurrent `rmtree`. A daemon reaper collects the worker while long-lived callers continue. A successful worker clears any stale cleanup-error artifact while it still owns the cleanup lock.
- Logs, preflight evidence, attempts metadata, and `final.txt` remain in the state directory after workspace cleanup. `attempts.json` is atomically rewritten after every completed model attempt, including exact-model-verified warmup entitlement evidence, so cancellation during a later entitlement fallback preserves the earlier model/category/effective-runtime evidence.

## Terminal States

- exit `0`: a non-empty terminal final artifact exists
- exit `75`: transient/capacity failure; retry only the same runtime/model if the parent policy permits
- other nonzero: blocked or failed; inspect `stateful status`, `attempts.json`, and bounded logs
- `stateful final` prints only the saved terminal artifact on success
- exit `127` with `fallback_workspace_retained: true` means preflight succeeded but the Codex executable was deterministically unavailable; use only that retained scope for the clean-context fallback, then run `stateful cleanup`
- workspace cleanup failure is terminal nonzero even when the reviewer produced a clean artifact; the error and retained state directory remain visible for recovery

Each attempt records runtime, requested/effective model, requested/effective effort when observable, category, exit status, and log paths. A warmup-derived entitlement attempt always has no final text and proves only model-chain fallback eligibility, not a repository review launch or result. For Codex, the helper resolves the emitted thread ID to its persisted rollout and requires matching `turn_context` model and effort before accepting the final artifact. Missing verification on a successful result is `runtime-unverified`; any observed model, effort, or permission mismatch overrides even an otherwise entitlement-shaped failed attempt and stops the lane. None of those conditions is entitlement evidence, so none enters the fallback path.

Claude Code publisher provenance is anchored to Anthropic's release-signing primary-key fingerprint `31DD DE24 DDFA B679 F42D 7BD2 BAA9 29FF 1A7E CACE`. For the exact detected version, the helper verifies `manifest.json.sig` over `manifest.json` from `downloads.claude.ai`, then requires the installed platform binary to match the manifest checksum. Native format and platform compatibility checks reject script/interpreter wrappers and incompatible artifacts but cannot distinguish every native launcher. macOS platform signing and notarization are possible future defense-in-depth checks, not current acceptance gates, and never substitute for the signed manifest. No Claude candidate may receive credentials or tracked review content until provenance and bounded capability probes pass.

## Deliberate Omissions

The helper no longer supports generic `auto` lanes, OpenCode, Cursor Agent, `gh-copilot`, `codex-parallel`, live working-tree snapshots, reviewer-visible Git shims, arbitrary child argv, report sinks, or legacy helper names. These surfaces caused ambiguous review counting, model drift, or unnecessary runtime code exposure.
