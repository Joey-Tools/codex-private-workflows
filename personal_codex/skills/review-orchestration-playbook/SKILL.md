# Review Orchestration Playbook

## Canonical Review Shapes

These are the only meanings of the named review shapes. Count completed logical reviewer lanes, never retries, model fallbacks, helper processes, or delivery gates.

| Requested shape | Required lanes |
| --- | --- |
| Single / single review / single internal review | One fresh-context Codex reviewer. |
| Double / double review / local double review | Single plus one actual Claude Code reviewer. |
| Triple / triple review | Double plus exact `@codex review` on an exact-host `github.com` PR and a complete terminal provider-authored GitHub Codex findings payload bound to the current PR head, whole-PR range, and isolated request. |

PR readiness means the effective review shape plus CI, unresolved-conversation, base/head, and merge-policy checks. It does not add a hidden reviewer lane.

### GitHub Codex fallback

GitHub Codex is the optional third lane, so its unavailability changes a requested triple review into an effective double review.

- Treat a missing PR, unsupported host or integration, unavailable GitHub Codex service, or unsupported operating identity as unavailable when directly known or proved by authenticated provider evidence. A missing response, timeout, generic request/HTTP failure, or guessed integration state is inconclusive rather than unavailable.
- The only supported host is exact `github.com`. Every other host is unsupported, including `sqbu-github.cisco.com` and every GitHub Enterprise host.
- PRs whose operating identity is in `{hoteng, hoteng_cisco}` are unsupported for this lane.
- Accept completion evidence only from provider-authored review/comment payloads with exact REST `user.login == "chatgpt-codex-connector[bot]"` and exact `user.type == "Bot"`. Require the selected review body plus every fully paginated associated inline review comment, or a terminal issue-comment body; missing or ambiguous payload, terminal nature, or association is `triple-inconclusive`. Exact `app.slug == "chatgpt-codex-connector"` check/run evidence can prove service start only when it is current-head and post-request. It never completes triple or proves a clean/no-findings result, even when `completed` / `success`; a same-App check may be unrelated and success may coexist with review findings. Unknown, missing, differently cased, or lookalike identities prove neither service start, terminal completion, nor authenticated no-start rejection.
- Report `requested: triple`, `effective: double`, and the concrete reason. Do not call the result a completed triple review.
- A blocked or inconclusive local lane is not a clean double merely because the GitHub lane was unavailable.

Read [pr-readiness.md](references/pr-readiness.md) for current-head evidence and the PR fix loop.

## Common Local-Lane Contract

Codex and Claude Code use the same frozen-scope and workspace scheme. Each logical lane receives its own workspace; lanes never share a checkout or reviewer context.

For a review-policy migration that changes this playbook, its reviewer profile, prompt templates, guard, compatible-version/provenance selector, stream validator/schema, or launcher, separate review subject from review control. Candidate-head Markdown, including the frozen repo-local copy at the review head, remains readable as review subject and scoped repository guidance, but the parent must source every executable or machine-interpreted review-control component from an independently trusted bundle pinned outside `base_sha..head_sha`. Record the bundle's absolute source path, publisher-provided release identifier or frozen commit, and canonical control-file manifest SHA-256 in the lane record; verify the same digest before each launch/spawn and after each lane. Never execute candidate-head Python or shell, or load a candidate-head machine schema, to bootstrap that candidate's formal review. If the active reviewer profile or any required Claude control component cannot be bound to the manifest, the migration review is blocked. Only when the prior trusted bundle lacks a compatible new guard may that prior trusted policy review the migration before merge and release, with the new guard activated only from the resulting trusted release. The guard is non-executable source: record and revalidate a trusted absolute Python interpreter, then launch it only with `-I -B -S` and the fixed clean environment defined by the contract. It must compile the manifest-bound runtime source bytes directly without adding the bundle to `sys.path` or accepting bytecode, native extension, or same-name module substitutes. The exact manifest algorithm, interpreter evidence, default three-module closure, and profile-specific version/capability/stream source and companion closures live in [review-lane-contracts.md](references/review-lane-contracts.md).

In this boundary, candidate-head Markdown is review subject and scoped guidance only; it never selects or executes the formal control plane.

The default guard profile keeps the exact three-source raw closure `review_runtime`, `review_runtime.common`, and `review_runtime.named_lane`; `preflight-claude` and `validate-claude-stream` use separate manifest-bound raw profiles for only their declared implementation/data dependencies. Companion revalidation repeats no-follow descriptor/type safety checks and compares the complete bounded content, not persistent file identity: a safe ordinary-file replacement with identical bytes is harmless. Guard-bound companion data is consumed from the immutable bytes captured by the initial validated read, never by reopening its path after final validation. These profiles never execute the env-shebang compatibility wrappers or use ordinary package import resolution.

1. Freeze an exact `base_sha..head_sha` range. Prefer a `wip/<topic>` branch and use a merge base when the target branch moved. Never synthesize a formal named lane from an uncommitted working tree or include untracked files. If the implementation checkout is dirty and the parent request already authorizes implementation or delivery mutations, create an intentional review-anchor commit on that review branch first. For a standalone report-only named review, do not create a branch or commit. When the intended review scope includes dirty or untracked state that no committed range represents, report review preparation as `blocked-authorization` and ask for an existing committed range or explicit authorization to create the anchor; use `blocked-input` instead for a clean checkout whose required range/PR/target selector is absent.
2. Materialize a separate lane-private Git workspace at `head_sha` only through the parent-recorded trusted guard's `materialize-worktree` subcommand. The resolved fixed-path Git executable is a declared parent trust root; before any safe `-c` override, the guard requires it to report version 2.45.0 or newer, without claiming to validate its publisher or on-disk identity. Older or unverified versions are `blocked-safety`. Before import, the guard requires the source worktree root, exact `.git` marker, and resolved admin/common/object directories to agree, then binds the marker by device, inode, file type, and owner and retains the resolved control-directory identities; a linked-worktree gitfile must keep both its forward `gitdir:` target and the admin directory's back-pointer bound to that exact marker. Control-file reads use no-follow, nonblocking descriptor opens followed by regular-file, owner, and stable-identity checks, and every storage revalidation rereads both linked directions. Marker `mtime`, `ctime`, and `nlink` are not identity or content checks, so benign churn in those fields is accepted while identity, type, owner, and forward/back targets remain exact. It rejects sibling `.bundle` / `.git` suffix discovery, initializes an empty owner-private destination, and exposes only the validated source object directory as a temporary read-only alternate under the destination's isolated configuration. The guard enumerates the exact frozen base/head reachable closure, hard-bounds object count, reachable logical bytes, head entry/blob/path bytes, and compressed pack bytes, imports only that manifest through bounded `pack-objects --no-use-bitmap-index` plus strict `index-pack`, removes the alternate environment, and proves the destination object inventory equals the frozen closure. The materializer and validator share one path-bearing Git raw-output envelope derived from 100,000 entries, 64 MiB of aggregate path bytes, and the 40-byte SHA-1 or 64-byte SHA-256 object-ID width; that derived ceiling bounds materializer `ls-tree` and validator `ls-tree`/`ls-files`/`status` output. Source configuration, hooks, and refs are never loaded into the destination; source/target ancestor discovery is fenced, ambient system/global config and attributes are disabled, no hardlinked or persistent alternate/promisor dependency or remote transport is allowed, and hooks, fsmonitor, submodule recursion, executable filters, and external diff commands stay disabled before repository initialization or checkout. It forces and persists Git-false commit-graph/multi-pack-index consumption, rejects source pack `.bitmap` before object traversal and target pack `.bitmap` at each private-storage audit, runs a bounded full object-validity `fsck` over the frozen endpoints, and rejects shallow, partial, incomplete, unexpectedly configured, or unsafe repositories before detached no-submodule checkout under the same sanitized direct argv. `git worktree add`, any clone/fetch/upload-pack path, a copied source config/hooks directory, and every pre-guard `git status`, `diff-files`, or `diff-index` are forbidden. If the trusted prior bundle lacks this subcommand during a self-policy migration, use that bundle's prior policy only for the bootstrap review, merge and release the migration, and activate `materialize-worktree` only from the resulting trusted release.
   Run the materializer with `GIT_NO_LAZY_FETCH=1` and `GIT_TERMINAL_PROMPT=0`, prove both endpoints and the range locally complete, and never let the reviewer trigger an on-demand fetch.
3. Immediately invoke the same trusted guard's `validate-worktree` on the exact materialized workspace/head, adding every applicable tracked guidance path. Its forced file-mode-aware ordinary/staged status is the first worktree-status operation. It rejects staged or ordinary dirt, `assume-unchanged` or `skip-worktree` hidden index bits, ignored artifacts, and materialized or initialized submodules; an absent or empty uninitialized gitlink remains valid, including the exact worktree-deletion status record caused by an independently verified absent gitlink. Any repository-visible direct `include.path` or `includeIf.*.path` key is `blocked-safety`, regardless of condition, target, contents, or a later apparent override. Includes stay disabled while the guard evaluates direct repository-local and per-worktree submodule path/activation values, including repeated `submodule.active` pathspecs for every raw gitlink without `.gitmodules` or a name/path mapping. Direct `core.fsmonitor` must be unset or Git-false; every direct `alias.*` key, executable clean/process filter, and external/driver/textconv diff command is rejected before that first status or reviewer inspection. The earlier materializer additionally rejects executable smudge filters before checkout; smudge-only configuration is relevant to materialization even though the post-materialization validator itself need not treat it as status-time execution. The validator allows stable in-worktree tracked source symlinks, rejects escaping or unstable/mismatched tracked symlinks, and requires every tracked `AGENTS.md` plus applicable tracked project guidance to be an ordinary in-worktree regular file. Symlink blob reads use one aggregate 30-second `git cat-file --batch` call, allow at most 4,096 tracked symlinks, cap each target at 16 KiB, and cap the batch output at 64 MiB. These guards compare only properties relevant to object completeness, checkout safety, clean state, or reviewer safety: `mtime`, `ctime`, and `nlink` churn is not launch identity or content evidence, and benign changes to those fields are accepted; ordinary file content is neither snapshotted nor rehashed. Every bounded Git/materialization/preflight/cleanup error is terminal `blocked` with reason `blocked-safety`; a cleanup failure must report the exact retained path. Do not launch the lane unless both subcommands succeed back-to-back and their evidence binds the same full head/workspace.
   Every validator Git call forces `core.commitGraph=false` and `core.multiPackIndex=false` and fences discovery at the worktree parent. The Claude process inherits the same ceiling and audited lane-local false values.
   Immediately before cleanup, require the control/worktree directory's recorded device, inode, and owner under the same owner-private parent. Detected identity drift is retained and reported. Defer forwarded signals through cleanup and the CLI receipt handoff: a pre-receipt signal rolls back the worktree, while a signal arriving during the complete flushed success receipt is post-handoff and cannot turn that receipt into a false failure. Do not overclaim the final pathname check-to-recursive-removal window against a non-cooperative same-UID replacement; stronger protection needs platform containment or descriptor-recursive deletion.
4. Enforce read-only reviewer behavior. The reviewer may use bounded read-only Git and source-inspection tools, but must not edit files, refs, the index, configuration, the PR, or external systems. The publisher-verified canonical Claude process may update ordinary CLI-owned authentication and runtime state in trusted real `HOME`, including credential refresh and possible cache or tool-result artifacts. These accepted CLI control-plane side effects are not model-authorized review mutations and do not authorize model/tool writes or deliberate host mutations; this contract does not enumerate or attest every CLI-owned `HOME` write. A filesystem read-only sandbox does not prove that state-changing MCP, Plugin, connector, or GitHub tools are absent, so the reviewer policy must forbid those actions and the parent must not authorize them. For Claude Code, native-sandbox `allowRead` is not a global host-read whitelist; the precise split between sandbox enforcement and prompt/model scope is defined below.
5. Start with no inherited conversation, parent findings, or other reviewer output. For Codex, use `fork_turns="none"`; on another platform use the equivalent zero-inherited-turn launch. Before launch, the orchestrator names the exact authoritative playbook path/version in the prompt: normally the active installed copy, or a frozen repo-local copy outside the candidate range. When a repository reviews a change to its own review policy, the independently trusted external bundle above is authoritative and the frozen repo-local copy at the review head is subject/scoped guidance only. Codex must load exactly the parent-named authoritative source before repository guidance and must never independently select another installed copy; a missing or mismatched source blocks the lane. Both local lanes follow the same discovery order: repository-wide `AGENTS.md`, changed-path metadata, applicable path-scoped `AGENTS.md`, repo-local domain skills, tracked project guidance, then hunks. Claude receives the trusted lane contract in its control prompt and reads only tracked Markdown guidance and repo-local skills from its worktree; neither lane executes candidate-head Python or shell as control-plane bootstrap.
6. Give the reviewer only review-control metadata: workspace path, `base_sha`, `head_sha`, exact range, exact authoritative instruction source/version, trusted-bundle absolute source/version/SHA-256 digest when applicable, instruction-loading order, read-only and evidence limits, review focus and non-goals, output contract, and—only for Codex—the exact sanitized Git argv prefix. The Codex prefix starts with `/usr/bin/env -i`, contains only the recorded trusted `PATH`, fixed `LANG`/`LC_*`, `PAGER`, and `GIT_*` allowlist including `GIT_CEILING_DIRECTORIES=<absolute-clean-worktree-parent>`, then the resolved trusted Git executable, fixed safe `-c core.commitGraph=false -c core.multiPackIndex=false` plus the remaining safe flags, and `-C <absolute-clean-worktree>`. It fixes `PAGER=cat` and `GIT_PAGER=cat` and passes `--no-pager`. Every Codex reviewer Git call must copy that token sequence exactly; bare Git, alternate executables/wrappers, reconstructed prefixes, changed safe flags, extra environment keys, and another worktree are forbidden. Every diff-producing command also passes `--no-ext-diff --no-textconv`. Do not prepare, paste, attach, or point either reviewer to a full diff, changed-file content, suspected finding, or generated diff file. This avoids front-loading a potentially million-character diff into the prompt.
   The fixed sequence contains the required safe `-c` flags; their token spelling and order are part of the supplied prefix rather than reviewer-selected configuration.
7. The reviewer discovers the change itself with bounded Git/tool calls: begin with counts, `--stat`, `--numstat`, and changed paths, then inspect one file, hunk, symbol, or test at a time.
8. Keep the reviewer's raw terminal result findings-only and preserve it verbatim. The orchestrator binds it to a separate lane record with three distinct layers: transport/schema trust as `artifact_status`, semantic disposition as `review_outcome`, and output shape as `presentation`, plus the exact range, runtime/model, workspace identity, and terminal state. A trusted artifact is not automatically clean. For Claude, require validator `classification: accepted` before passing the unchanged `result` through the manifest-bound trusted-bundle `classify-review-result` guard profile backed by [`review_result.py`](scripts/review_runtime/review_result.py); prose inspection supplies only its explicit `content_assessment` and never replaces or repairs artifact validation. `canonical-clean` means the entire result is exact `No findings.` after removing outer ASCII whitespace only. `extended-clean` means a concise non-actionable positive/coverage summary followed by a final nonempty logical line that is exactly `No findings.`. An actionable finding takes precedence over that terminal sentinel and yields `review_outcome: findings`, `presentation: contradictory`; uncertainty, semantic conflict, or content that cannot be confirmed yields an undetermined outcome and ambiguous presentation. The raw result is never normalized or overwritten. The orchestrator may add commands, tests, or residual risk only when independently observable; never force that metadata into the findings-only result. Intermediate reasoning, tool traces, keepalives, and partial output are not review evidence.
9. Remove the lane worktree after the terminal artifact has been collected, unless a precise recovery reason requires temporary retention.

Repository-approved preflight may validate workspace containment, frozen-scope identity, object completeness, or reviewer-sandbox safety before launch. A tracked secret delta is not a reviewer-launch gate: the trusted reviewer may inspect the original tracked diff and necessary tracked context, including repository secrets, without redaction or rewriting. Reviewer/runtime authentication credentials, untracked files, unrelated repositories, broad workspace dumps, and home-directory content remain outside the consented scope. Keep private control evidence outside the reviewer-visible workspace, and never turn a preflight into a hidden reviewer lane or inject a generated full diff into the prompt.

Read [review-lane-contracts.md](references/review-lane-contracts.md) before launching a lane.

## Codex Lane

- Run `materialize-worktree` and then `validate-worktree` through the parent-recorded absolute trusted guard path on the same exact frozen worktree/head immediately before spawn; require the materializer receipt and let only the validator perform the first status query.
- Spawn the dedicated `reviewer` role with `fork_turns="none"`; do not use a default coding agent, an inherited-context child, or a parent-thread continuation.
- Give it the clean Git worktree and the common lane inputs only, including the exact sanitized Git argv prefix. The agent loads applicable Markdown skills and project instructions, then inspects the frozen range without a bare Git invocation; all diff-producing commands explicitly disable external diff and textconv.
- Use the configured `gpt-5.6-sol` with `xhigh` reviewer profile. If that profile is deterministically unavailable, the required Codex lane is blocked; transient failure is inconclusive. Do not silently select another profile.
- The existing supplied-diff/private-Git Codex helper is not this lane and does not satisfy single review.

## Claude Code Lane

- Double and triple review require an actual Claude Code process in a second independently materialized clean Git workspace under the common pre-status-isolation/read-only/no-prepared-diff contract.
- The canonical Claude Code compatibility range is `>=2.1.211,<3.0.0`, defined once in [`scripts/review_runtime/claude_version_policy.py`](scripts/review_runtime/claude_version_policy.py). Do not duplicate that range or turn an audited patch into a global eligibility pin. Claude Code `2.1.212` is the audited per-version stream-schema baseline, not a global eligibility pin.
- Before any prompt, credential, repository, range, PR, or review-workspace input reaches Claude, invoke `<trusted-python-absolute-path> -I -B -S <trusted-bundle-absolute-path>/skills/review-orchestration-playbook/scripts/named_lane_guard preflight-claude`. This formal subcommand raw-loads the manifest-bound implementation behind [`scripts/named_claude_preflight`](scripts/named_claude_preflight), its version/capability/stream-contract dependencies, and their exact companions from the trusted bundle without ordinary package import resolution; direct execution of that env-shebang wrapper is low-level compatibility only and cannot satisfy a named lane. For this formal profile, `$HOME` means the guard-derived canonical home of `pwd.getpwuid(os.getuid())`; the clean launcher does not inherit ambient `HOME`, and an unresolvable account home fails closed. Its candidate order is an authoritative explicit absolute override, the highest compatible stable side-by-side install under `$HOME/.local/share/claude/versions/`, then the controlled active-install paths `$HOME/.local/bin/claude`, `/opt/homebrew/bin/claude`, and `/usr/local/bin/claude`; caller `PATH` is ignored. Candidate presence is tri-state: only exact absence may advance priority, while I/O, enumeration, resolution, or identity uncertainty stops fail closed. Before executing any candidate, require an in-range declared version, a supported native executable, and exact signed per-version manifest size/SHA verification; publisher verification returns a descriptor-bound strong source identity including `ctime`. Materialize a private digest-verified executable snapshot only while that identity remains stable, run bounded credential-free `--version` and mandatory `--help` probes against that snapshot with empty stdin, reverify the snapshot after each probe, then descriptor-rehash and rebind the mutable source before acceptance. The help probe verifies only the advertised capability surface, not launch semantics, final argv, final merged sandbox, managed permission arrays, or path-rule evaluation. The preflight emits one bounded JSON object and never downloads or installs Claude Code, changes an active symlink, or executes a script/wrapper, mutable install path, or caller-`PATH` candidate. Only `classification: accepted` with `reason: compatible-version-selected` supplies `<resolved-compatible-claude-path>`. Persist that result in a parent-private, current-user-owned, single-link regular file outside the model-visible worktree with no group or world permission bits; bind the selected version, signed artifact, descriptor identity, capability status, and compatibility-profile/audited-baseline/capability-contract digests for launch and validation. Deterministically unavailable compatible selection is `compatible-version-unavailable`; an unsupported declaration is `unsupported-version`; coherent signed/declared/observed disagreement is `signed-version-identity-mismatch`; an advertised-surface mismatch is `capability-contract-mismatch`; publisher failures remain `publisher-verification-failed`; and dependency, network, snapshot, probe, stream-contract, or identity uncertainty remains inconclusive. Never collapse uncertainty into deterministic unavailability, continue past an uncertain higher-priority candidate, substitute another provider, or downgrade the requested shape.
- Acquiring a missing compatible release is a separate, explicitly authorized host-mutation workflow through the official installer/version manager. A named review request alone does not authorize installation, downgrade, changing the active Claude version, or repairing symlinks; do not install or downgrade to `2.1.212` merely because it is the audited stream baseline.
- The Claude process must start fresh and must not receive the Codex artifact or parent findings.
- In the accepted real-`HOME` native-sandbox design, the detached worktree is the review scope while real `HOME` remains the trusted CLI control plane. `Read`, `Grep`, `Glob`, and sandboxed `Bash` may be available. Launch must request global `denyWrite` and critical-sensitive-root `denyRead`; those requested controls define the native-sandbox enforcement boundary, while selected `allowRead` is not a global host-read whitelist. The prompt/model contract, not an OS-wide read allowlist, forbids reading outside the detached worktree.
- If Claude Code reports that command output was persisted or spilled to a path outside the detached worktree, treat that report as a possible trusted CLI control-plane write only. Claude must not follow it with `Read`, `Grep`, or `Glob`; it must rerun a narrower bounded command over worktree paths. An observable structured tool path outside the exact worktree, including a symlink escape, adds deterministic blocked evidence. The final machine classification still follows the validator's global precedence, so concurrent malformed or otherwise inconclusive stream evidence yields `inconclusive` with both reasons rather than accepted findings.
- Capability output and `system/init` evidence—including Claude Code 2.1.212 baseline output—cannot prove the final merged native-sandbox settings, managed permission arrays, or path-rule evaluation. Record those sandbox controls as requested configuration, never as independently verified effective enforcement.
- After successful `validate-worktree`, pass the mandatory parent-private accepted `--preflight-result` to the same parent-recorded absolute trusted guard's process-only `run-claude` supervisor. The guard binds `<resolved-compatible-claude-path>` to that evidence, opens the non-symlink source executable, checks the relevant device/inode/type/mode/owner/size policy, copies and SHA-256-verifies the signed bytes into a private guard-created mode-`0500` snapshot, and launches that snapshot as its direct child `argv[0]` with direct argv/no shell. Snapshot copy and rehash consume the same monotonic deadline. Snapshot creation/handoff and later cleanup are separate forwarded-signal-masked critical sections; between them the bounded supervisor owns structured signal forwarding and process cleanup. Cleanup always acts relative to the retained output-parent descriptor and recorded snapshot-leaf identity; lexical parent revalidation remains an output-publication gate and never blocks safe snapshot cleanup. The formal CLI reports incomplete cleanup as structured `inconclusive` / `snapshot-cleanup` evidence with exact `process_reason` plus either exact `retained_path` while the lexical parent binding is revalidated or descriptor-bound `retained_locator` device/inode/leaf evidence after lexical drift. Source `mtime`, `ctime`, and `nlink` are not launch identity or content checks; benign churn is accepted, while size/SHA-256 or relevant identity/policy drift fails closed. Replacing the raw source path after the snapshot is bound cannot change the bytes executed, and the parent must not substitute before/after raw-path checks for this launch binding. The process receipt records `launch_binding` with mode `verified-snapshot`, preflight digest, resolved source path, relevant identity, and signed artifact size/SHA-256. Claude still starts fresh from the clean worktree with a non-persistent session, inline native-sandbox settings that explicitly disable hooks and bundled skills, only `Read`/`Grep`/`Glob`/sandboxed `Bash`, no MCP/browser/edit/write/web/task tools, and the bounded control prompt on stdin. Its EOF read and bounded worktree Git resolution share the 1,800-second monotonic deadline with process supervision, and stdout and stderr are capped at 64 MiB each (128 MiB aggregate). Test-oriented timeout, stream, and prompt overrides may equal or tighten—but never raise—the production caps of 1,800 seconds, 64 MiB per stream, and 256 KiB; the direct Python API enforces the same ceilings.
- Before invoking `validate-claude-stream`, the parent must compare the receipt's exact `launch_binding.preflight_sha256`, `resolved_path`, `identity`, `artifact_sha256`, and `artifact_size` with the accepted preflight and the expected launch binding. The validator separately rereads `--preflight-result`; it does not consume or validate `launch_binding`.
- For each private output path, the caller supplies a lane-unique, current-user-owned mode-`0700` canonical real parent directory outside the worktree and an absent, non-symlink leaf; `run-claude` keeps a descriptor for that validated directory across launch and anchors publication and rollback to it. Publication, rollback, and the complete flushed `launch_binding` process receipt remain signal-masked through one explicit CLI commit point. A signal observed before that receipt commits, or a receipt write/flush failure, removes the complete output pair before propagation; a signal after the flushed receipt cannot turn it into a false failure. Cleanup checks the creating write's `(st_dev, st_ino)` and preserves identity drift already observed before unlink. Because POSIX/Python has no portable conditional unlink, this lightweight lane relies on cooperative same-UID exclusivity for that private directory; a non-cooperative same-UID replacement in the final check-to-unlink window is outside its guarantee.
- The launch environment is rebuilt from an explicit allowlist: `HOME`, `USER`, `LOGNAME`, and `SHELL` come from the current account's `pwd` record; `PATH` is trusted; only allowlisted locale/UI, proxy, and CA variables may pass through. Ambient `NODE_EXTRA_CA_CERTS` remains excluded unless the caller explicitly opts in with the value-free `--inherit-node-extra-ca-certs`; the configured value must then name an exact absolute readable non-symlink regular file that passes a stable no-follow identity check. This direct-lane path is host control-plane compatibility rather than copied or attested helper CA material. Git is forced to use no lazy fetch, prompt, replacement objects, global/system config, or optional locks. Ambient Claude/Anthropic, cloud-provider, dynamic-loader, and tool-control variables are not inherited. Canonical provenance rejects npm/NVM shebang shims and all other scripts/interpreter wrappers; `run-claude` never expands trusted `PATH` to accommodate them, and guard supervision does not establish executable provenance. This remains the canonical direct actual-Claude process—not `isolated_review` or a helper reviewer.
- Treat `run-claude` only as the launch guard and bounded process supervisor. It proves neither compatible-version provenance nor structured-output validity. Only complete raw stdout collected after successful TERM/KILL/drain/reap cleanup of the initial supervisor process group and inherited streams may proceed to output validation. Every `run-claude` supervision failure—including timeout, either-stream overflow, drain/reap failure, residual members of that group, or inherited-stream leaks—is `inconclusive`. This process-only supervisor is not a process-tree sandbox: a descendant that deliberately uses `setsid()` or `setpgid()` and closes inherited streams is outside its cleanup guarantee, so do not claim whole-process-tree quiescence.
- Independently require a strict `stream-json` envelope with exactly one leading `system/init` event as the first nonblank record and one trailing terminal `result` event as the last nonblank record. Bind `claude_code_version` to the exact preflight-selected release, select the closed `legacy-base` profile for `>=2.1.211,<2.1.216` or closed `extended-2x` profile for `>=2.1.216,<3.0.0`, and validate every admitted intermediate record against that profile's closed, session-bound contract. Fail closed when either envelope event is missing, duplicated, malformed, or misordered, when an intermediate record violates the selected profile, or when required init evidence does not match the exact clean-worktree cwd, `dontAsk`, the exact `Read`/`Grep`/`Glob`/`Bash` tool set, empty MCP/slash-command/skill/plugin surfaces, requested model, exact publisher-verified CLI version, and local-login authentication source. The init object uses the selected profile's closed top-level allowlist; any unrecognized field is inconclusive. These observable fields verify only the reported runtime surface; they do not prove the final merged sandbox, managed permission arrays, or path-rule evaluation.
- For `named-direct`, the validator gates required absolute `Read.file_path`, present absolute `Glob.path`, present absolute or cwd-relative `Grep.path`, and required `Glob.pattern`; a relative `Grep.path` is anchored to the exact bound cwd and checked lexically plus through symlinks, while every tilde-prefixed shorthand remains `scope-unverified`. An omitted `Glob.path` uses the exact cwd, while an omitted `Grep.path` is not promoted to validator-proven cwd semantics. `Glob.pattern` accepts a bounded, provable relative subset, normalizes leading `./` only, and rejects absolute, exact-parent, home-shorthand, extglob, escaping alternative, or unsupported syntax as blocked or `scope-unverified` according to what containment can prove. Dynamic directory components are checked with a conservative entry/state/depth-bounded validation-time traversal; any reachable external symlink is outside evidence and any exhausted scan is `scope-unverified`. The validator opens one cwd descriptor at validation start, reuses that binding, and rechecks it before returning. ## GitHub Codex Lane

- Use only a PR on exact host `github.com` whose operating identity is not in `{hoteng, hoteng_cisco}`. Every other host, including every GitHub Enterprise host, is unsupported.
- Request the lane with the exact `@codex review` PR comment after the frozen head is current.
- Posting the request comment is not completion. An authenticated provider rejection may prove that no run started and the integration/service is unavailable only when it is tied to the exact request/dispatch or satisfies the sole-unresolved fallback; a full SHA alone does not bind a same-head no-start response to one request. An acknowledgement, run, or review activity proves service start.
- Count provider evidence only when its exact REST bot identity or exact app slug matches the accepted identity above. Unknown or lookalike authors/apps make the lane inconclusive rather than proving rejection, start, or completion.
- Bind the accepted findings payload to the current PR head and whole-PR range under the request-isolation contract in [pr-readiness.md](references/pr-readiness.md): allow at most one acceptable exact request per unchanged head and never post a second one. Server timestamps prove ordering, not request/run lineage; review/comment evidence with no request/run identifier is triple-inconclusive whenever an older request might overlap. Fetch a selected review's body and every fully paginated associated inline review comment, or the terminal issue-comment body, and require an unambiguous complete terminal outcome. An exact-App current-head check/run with non-null `started_at` strictly later than the request is service-start evidence only, regardless of terminal status or conclusion. Any code change invalidates earlier GitHub Codex evidence and permits at most one new-head request/result.
- At the first selected-PR range freeze, persist immutable parent-owned `range_origin` with exact `kind`, `base_sha`, and `head_sha`; `kind` is only `caller-supplied` or `pr-derived`, and original caller endpoints remain immutable. If `pr_merge_base` changes after the request while `pr_head_oid` stays unchanged, invalidate every previously counted whole-PR artifact and never post a replacement request on that same head. Apply [base-only-retarget-state-machine.json](references/base-only-retarget-state-machine.json) before generic scope mismatch: a missing origin or stale range stops before local lanes; a caller-supplied current range may recover local lanes only when the caller supplied it, while a previously PR-derived range may recover them through normal current-range rederivation. Those recovery passes run the local lanes but do not unblock the same-head GitHub lane. Report readiness `blocked-input` (`base-changed-same-head`) and `requested: triple`, `effective: triple-inconclusive` until a separately authorized ordinary change creates a new head. Do not manufacture an empty or anchor commit solely to regain eligibility.
- If the lane is unavailable, apply the explicit triple-to-double fallback above. If it ran and reported findings, the lane is available and its findings must be handled; findings are never an unavailability reason.
- If an existing supported PR's current `headRefOid` does not equal the frozen `head_sha` and the parent did not authorize publishing or changing the PR head, do not mutate the PR. Report `requested: triple`, `effective: triple-inconclusive`, and GitHub lane status `blocked-authorization`; a PR/head mismatch is not an availability fallback.
- For every selected PR, including an explicitly named one, require authenticated lifecycle `state == "open"`, `merged == false`, and `merged_at == null` before local PR-specific lanes or the GitHub lane. Missing/contradictory lifecycle evidence is `blocked-input` (`pr-lifecycle-unverified`) plus triple-inconclusive; closed-unmerged is `selected-pr-closed` and may reduce only an otherwise fully scoped, never-started third lane to effective double; merged is terminal `already-merged` / `selected-pr-merged`. Revalidate at selection, before posting, before accepting a result, and before readiness/merge. An observed non-open lifecycle at any mandated snapshot after request/service start invalidates evidence and stays triple-inconclusive. These point-in-time snapshots do not prove that no intermediate close-and-reopen occurred between them.
- If a supported service started but its provider-authored findings payload, terminal nature, head/range/request association, or pagination completeness is missing or ambiguous, report `requested: triple`, `effective: triple-inconclusive`; do not convert that uncertainty or check-only evidence to effective double, completed triple, or a clean result.

## Workflow

1. Classify the request.
   - A review-only child that explicitly forbids orchestration inspects its assigned range and returns findings only. It must not start other reviewers, edit code, wait for CI, or mutate the PR.
   - Resolve a standalone named review deterministically: preserve an explicit frozen range for local lanes; independently select an explicitly named PR or exactly one open PR associated with the exact current head repository/branch when PR-specific/triple work needs one. An explicit-range-only single/double is fully scoped locally and requires no PR probe. More than one required PR candidate leaves the GitHub/PR-specific lane `blocked-input` until the caller names a PR; a frozen range does not select among them, although fully scoped local lanes may run. An authenticated complete zero-candidate lookup proves the no-PR path, but does not supply a local review range: require an explicit committed range or explicitly named target/base from which to freeze `<merge_base>..HEAD`; never guess the target/base. For triple, detached HEAD or unknown head ownership without an explicit PR cannot prove no PR: run the scoped local lanes, but report the GitHub lane `blocked-input` and `effective: triple-inconclusive`, not effective double. Once a PR is selected, independently validate its current base/head metadata and unique merge base; a caller-provided range does not become whole-PR scope merely because its head matches.
   - A standalone named review request is report-only unless Joey also asks to fix or deliver the change. It does not authorize branch creation, an anchor commit, push, PR creation, or PR branch/metadata changes. A clean checkout with a missing selector is `blocked-input`; reserve `blocked-authorization` for intended dirty/untracked state that would require an unauthorized anchor commit. Bare triple additionally authorizes only the scoped `@codex review` request on an already-existing supported PR. If no PR exists and an explicit committed local range is available, run the requested local lanes and reduce requested triple to effective double. If a PR exists but its host, identity, integration, or service is directly known unavailable, retain the existing-PR head-alignment preflight before running the effective double. The parent prepares the requested lanes and returns findings; it does not edit code, start delivery gates, or enter a fix loop on its own.
   - A PR/full-workflow request with no named shape defaults to single review.
2. Preserve any parent-provided frozen range before consulting PR state. Otherwise derive it only from the explicitly or uniquely selected PR's authenticated base/head metadata and unique merge base, or from an explicitly named no-PR target/base; a missing or ambiguous selector is `blocked-input`. For every selected existing PR, persist immutable parent-owned `range_origin.kind`, `range_origin.base_sha`, and `range_origin.head_sha` at its first range freeze; only `caller-supplied` and `pr-derived` are valid kinds, and a missing or ambiguous selected-PR origin is `blocked-input` (`range-origin-unverified`). For every selected existing PR in a PR/full-workflow request or any standalone named review request, independently read `baseRefName` as `pr_base_ref`, `baseRefOid` as `pr_base_oid`, and `headRefOid` as `pr_head_oid`; with lazy fetching disabled, require both endpoint commits locally and require `git merge-base --all pr_base_oid pr_head_oid` to return exactly one full `pr_merge_base`. Missing/ambiguous metadata, objects, or merge-base results are `blocked-input` (`scope-unverified`). Before applying the generic same-head/different-base `scope-mismatch` branch, inspect the parent-owned audit record for an accepted request on `pr_head_oid` and narrowly revalidate that exact request. If its recorded request-time merge base differs from current `pr_merge_base`, apply the dedicated state machine above: a stale or unauthorized range stops before local lanes, while an exact current-range recovery with authorized provenance proceeds to local lanes but leaves the GitHub lane blocked. Never fall through to the generic branch. Otherwise a selected PR's explicit frozen range satisfies PR-specific readiness or triple completion only when `base_sha == pr_merge_base` and `head_sha == pr_head_oid`. A same-head/different-base range is `blocked-input` (`scope-mismatch`): preserve the caller's range, do not silently rewrite it, do not start or count PR-specific lanes from it, and never describe its local review results as whole-PR coverage. Then record `pr_head_oid` separately and compare it with the intended `head_sha` before creating or running any local lane or consuming PR CI/conversation/readiness state. This preflight applies to a selected PR in single, double, triple, and triple already reduced to effective double; explicit-range-only standalone single/double with no selected PR and the proven no-PR path have no PR-head comparison or selected-PR range comparison. On head mismatch, publish/freeze the intended head only when PR mutation is separately authorized; otherwise leave the PR unchanged and report readiness `blocked-authorization`. A still-eligible triple candidate also reports `effective: triple-inconclusive` with GitHub lane status `blocked-authorization`. For the same mismatch on an already unsupported PR, keep `requested: triple`, `effective: double`, and report readiness `blocked-authorization`; do not treat the mismatch as making the already-unavailable lane triple-inconclusive or as permitting readiness to continue.
3. Freeze the exact range; for each requested local lane, run trusted `materialize-worktree` and then `validate-worktree` so no candidate/ambient hook, fsmonitor, filter, or status operation runs before the isolated checkout safety gate.
4. Run the fresh-context Codex lane.
5. For double or triple, run the separate Claude Code lane.
6. For triple, classify directly known no-PR/host/identity unavailability before request. On an otherwise eligible PR whose frozen range is exactly its current `pr_merge_base..pr_head_oid`, enforce one acceptable exact `@codex review` request per unchanged head: reuse the recorded request and never post a second one; overlapping or unassignable results are triple-inconclusive. Immediately before accepting a complete terminal provider-authored findings payload, re-read the PR base/head OIDs, recompute the unique merge base, and require that exact equality again; a changed head or merge base invalidates whole-PR lane evidence. If the merge base alone changed after the one request, do not follow the general rerun rule for the GitHub lane: use the explicit `base-changed-same-head` blocker above. Fetch the selected exact-bot review body plus every fully paginated associated inline review comment, or a terminal exact-bot issue-comment body, and fail `triple-inconclusive` when its payload, terminal nature, or association is missing or ambiguous. Unknown pre-request integration/service status does not block the one request. An authenticated provider rejection may prove no-start integration/service unavailability and records effective double. Posting the request is not service start; missing response or generic failure is triple-inconclusive. An exact-App current-head post-request check/run proves service start only, never triple completion or no findings, even when `completed` / `success`. Once acknowledgement or run/review activity proves start, malformed, stale, ambiguous, incomplete, or check-only evidence is triple-inconclusive, never fallback.
7. Only when Joey's requested fixes, delivery, or PR orchestration, apply actionable findings in the parent implementation workspace, rerun affected tests, freeze the new head, and rerun every requested lane invalidated by the fix. A non-canonical or ambiguous presentation alone never triggers a rerun. Rerun only when range/head evidence became invalid, a finding fix produced a new head, or Joey explicitly requested another review; otherwise report the ambiguous disposition as a decision point.
8. For PR readiness, complete the remaining CI, conversation, base/head, and merge-policy checks in [pr-readiness.md](references/pr-readiness.md).
9. Report requested shape, effective shape, exact range/head, and each lane's runtime/model, `artifact_status`, `review_outcome`, `presentation`, verbatim raw result, fallback reason if any, and cleanup state. Keep a non-accepted artifact's outcome and presentation not evaluated rather than collapsing transport failure into semantic findings or clean.

## Egress Consent

Any unambiguous request classified as a named review shape authorizes scoped tracked-code review egress for exactly that shape. Examples include `single review`, `single code review`, `单重 review`, `单一 review`, `double`, `double review`, `double code review`, `双重 review`, `triple`, `triple review`, and `三重 review`:

- Single authorizes OpenAI Codex.
- Double additionally authorizes Anthropic Claude Code.
- Triple additionally authorizes, when supported, current-head GitHub Codex.
- No named shape authorizes a substitute external reviewer.

The selected reviewer is a trusted processor. Consent includes the named repository's original tracked diff and necessary tracked context, including tracked repository secrets; do not redact, rewrite, or suppress reviewer-visible tracked content because a secret audit is blocked, inconclusive, or violating. It excludes automatic discovery of reviewer/runtime authentication credentials, untracked files, unrelated repositories, broad workspace dumps, and home-directory content.

Read [egress-consent.md](references/egress-consent.md) before external egress. Approval justifications must name the exact repository, frozen range, destination, included tracked-code scope, tracked-secret handling, and exclusions.

## PR/Master Secret Admission

Secret admission is independent of every named reviewer lane and never delays or blocks reviewer launch.

- For each countable candidate, use its exact raw byte value as the key and count it globally over the complete base and head tracked trees: raw Git path bytes, regular-file blob bytes, and symlink-target bytes. Gitlink entry paths count; gitlink object IDs and submodule content do not.
- Permit `head_count <= base_count`. Equality, deletion, and movement across paths, blob/symlink surfaces, modes, or byte offsets pass. Only a first appearance or global count growth blocks admission.
- Do not derive Base64, hex, URL-encoded, escaped, hashed, or other transformed variants. This is a deliberate detection limitation; a transformed byte string matters only when it independently becomes an exact scanner candidate.
- A non-exact dynamic expression does not enter the counter. Incomplete tree enumeration, unreadable bounded objects, or lost count integrity makes admission `inconclusive`.
- Report only head-side added locations for positive-delta candidates; omit unchanged occurrences. Use one-based lines for complete text evidence, `line: null` for new-path or binary fallback, and line `1` for symlink targets. Mark incomplete mapping as `location_status=inconclusive`; when endpoint trees cannot distinguish a moved occurrence from an identical new copy, omit the ambiguous locations instead of relying on heuristic rename attribution.

Run required PR/master admission directly with `isolated_review secret-admission --repo <repo> --base-ref <base_sha> --head-ref <head_sha>`. It scans the immutable Git trees under `review_contract: admission-only-no-reviewer` and does not create reviewer state, materialize a review workspace/diff/prompt, or start any provider. Exit `0` plus `temporary_cleanup_status: complete` is `clean`; exit `1` is proved violations and remains `1` even if later location mapping or temporary cleanup is incomplete; exit `75` is an inconclusive scan or a clean scan whose temporary cleanup failed. A head change invalidates the result. If an explicitly requested low-level helper run separately exists, its compatible `stateful final` and `stateful admission` results remain independent diagnostics, but neither is the required admission producer.

## Low-Level Helper Boundary

The `isolated_review` helper retains a frozen supplied-diff runtime backed by a helper-owned detached worktree and private minimal Git database for low-level compatibility and Claude runtime-security work. It records `review_contract: supplied-diff-private-git` and `named_lane_eligible: false`. Clean committed-head content is the default; only explicit `--include-source-wip` may capture staged, unstaged, and nonignored untracked content as a digest-bound diagnostic snapshot, and WIP evidence cannot satisfy PR-readiness or merge-ready exact-commit gates. Its workspace and prompt contract differ from the canonical named shapes above.

- Do not use its Codex path to satisfy single review.
- Do not count a supplied-diff helper run as the Claude Code lane of a named double/triple review.
- Do not add helper preflight, fallback, or retry attempts to the review count.
- For helper-only Claude diagnostics, use `explicit-claude-review`; use `explicit-claude-with-copilot-fallback` only after separate user consent explicitly authorizes both destinations. Named `double`/`triple` phrases and legacy `double-review`/`triple-review` values are not low-level helper consent markers.
- For helper authentication, apply precedence `ANTHROPIC_API_KEY` > `CLAUDE_CODE_OAUTH_TOKEN` > local login and opaque-forward only the winning explicit value. An explicit API key or OAuth token bypasses helper local-login carrier access and its refresh transaction. For a catalogued local-login artifact, acquire the certified host refresh locks before the first credential read and retain one outer transaction lease through credential exposure, reviewer/broker quiescence, rotation persistence, and final verification. Release only after quiescence and persistence are proven; otherwise retain descriptor-bound stale-fence evidence and pause without automatic deletion or Copilot fallback.
- Intentionally retained shared refresh-lock directories never authorize a lexical recovery or cleanup pathname; report only descriptor-bound residue.
- The low-level helper accepts publisher-verified Claude Code versions `>=2.1.211,<3.0.0`, but local-login refresh writeback additionally requires an exact signed artifact in its credential-lock catalog. An uncatalogued in-range artifact therefore blocks only helper local-login; helper API-key/OAuth modes and the named direct lane remain governed by the full compatible range. The helper does not run an OAuth freshness warmup: an expired local-login access token may refresh inside the isolated carrier, while `Login expired`, HTTP 401, or refresh failure pauses and directs Joey to reauthenticate. Authentication failure never authorizes Copilot.
- Pass helper Claude raw stdout through the same preflight-bound strict validator used by the direct lane. Preserve its machine classification exactly; broker/carrier success, prose inspection, or partial output never supplies findings or authorizes fallback.
- Read [helper-contract.md](references/helper-contract.md) before modifying or debugging the helper.
- Read [claude-runtime-trust.md](references/claude-runtime-trust.md) before changing the helper's version, provenance, platform, sandbox, capability-probe, carrier, or refresh-transaction behavior.

## Guardrails

- Do not precompute a full diff for a named local lane, even when it seems convenient or the change is small.
- Do not run formal review against a live dirty working tree. Create an explicit review anchor only when the parent request authorizes that mutation; otherwise stop as `blocked-authorization`.
- Do not claim clean review without a trustworthy terminal artifact for every required lane in the effective shape.
- Do not treat validator acceptance as a clean outcome, normalize the raw result, or let prose inspection substitute for artifact validation.
- Do not automatically rerun a lane because its presentation is extended, contradictory, ambiguous, or otherwise non-canonical. Rerun only for invalidated range/head evidence, a finding fix on a new head, or an explicit request.
- Do not treat an exact-App check/run as the GitHub lane's terminal artifact or no-findings payload; its role is service-start evidence only.
- Do not report a requested triple as completed triple after GitHub Codex fallback.
- Do not silently replace Claude Code with another provider.
- Do not downgrade a model for capacity, timeout, network, or other transient failure.
- Do not infer entitlement from silent model substitution.
- Do not follow a CLI-reported persisted/spilled-output path outside the Claude worktree. Use absolute `Read.file_path` and present `Grep.path` / `Glob.path` values; keep required `Glob.pattern` relative, parent-free, home-shorthand-free, and extglob-free. Ordinary recursive, intermediate-component, character-class, and bounded simple brace patterns such as `**/*.py` and `src/**/*.{py,md}` are allowed, as are leading forms such as `./**/*.py`; intermediate `.` is not. Rerun a narrower bounded worktree-scoped command; an observed structured outside-workspace tool read invalidates the lane.
- Do not start another reviewer from a findings-only review child.
- Do not use state-changing MCP, Plugin, Git, or GitHub actions inside a review-only lane.
- Do not block trusted reviewer launch on a tracked secret delta or redact tracked reviewer input based on admission status.
- Do not count encoded variants for admission; enforce only the complete-tree global exact-raw-value non-growth rule.
- Do not require or start a low-level helper reviewer to produce PR/master admission; use the direct admission-only command. Do not treat `stateful final` as secret-admission evidence or merge the two terminal decisions when helper compatibility state exists.
