# Remote Host Context

## Overview

Use this skill when the answer depends on where Joey worked, not just which local repository is open.
It standardizes a small read-only SSH preflight across Joey's default remote hosts before summarizing activity, judging missing evidence, or recovering recent context.

## Default Host Policy

- Treat these hosts as Joey's default evidence scope unless the user explicitly narrows it:
  - local machine
  - `BL-mac-mini-m4-hoteng`
  - `miku-bot-dev` (Joey may also refer to it as `miku-server-dev`)
  - `hoteng-srv-01`
  - `codex-hoteng-srv-01`
- Do not require Joey to mention any default remote separately in future conversations. Include all four SSH aliases in the default preflight.
- Treat `hoteng-srv-01` and `codex-hoteng-srv-01` as distinct evidence roots even though they reach the same remote hostname; they use different login users and Codex homes.
- For tasks rooted in local mutable state such as Apple Notes, local GUI apps, or local databases, keep all writes local. Remote hosts contribute evidence only unless Joey explicitly asks to modify them.

## Workflow

1. Start with a read-only SSH preflight.
- Check each remote host for reachability, user/home identity, and presence of `~/.codex`.
- If the task will need text search inside remote history, also check whether `rg` is available on that host.
- Report the exact missing gate: SSH/auth failure, host unreachable, missing `~/.codex`, or no matching evidence in the requested date range.
- Prefer the installed helper path `/Users/hoteng/.codex/skills/remote-host-context/scripts/remote_codex_probe.py` when available, so approved installed-helper prefix rules match; use the checked-out skill helper only for local development or when the installed helper is absent.
- Treat that preflight as a bounded host check, not a license to keep host-specific `jq`, `rg`, or shell-wrapped remote queries in `~/.codex/rules/default.rules`.
- If the same remote summary or mining pattern starts repeating, factor it into a dedicated helper under this skill instead of widening bare `ssh` approvals with more literal commands.

2. Use the helper for bounded remote Codex reads once the preflight is clear.
- Prefer `python3 /Users/hoteng/.codex/skills/remote-host-context/scripts/remote_codex_probe.py preflight ...` for recurring host checks.
- When Joey supplies a `codex://threads/<id>` URL, do not call `read_thread` unless the service supports and the caller sets server-side accepted item-type filtering, an item-count limit, and a whole-response byte cap of at most 32 KiB (32,768 bytes) on the encoded response. Require those three controls alongside one exact thread per call, `turnLimit: 1`, `includeOutputs: false`, and `maxOutputCharsPerItem` at most 400; the service itself must emit only selected thread metadata plus at most 12 short user or agent message snippets. The 32 KiB cap is an upper bound, not a target, and must be enforced before any payload reaches the caller. Caller-side projection or truncation after receipt does not qualify.
- If any whole-response control is unavailable, as in the observed API, skip `read_thread` entirely. Invoke `session-meta` directly only when the creation date is known, scan only that creation-date directory (including the bounded UTC/host-local variants of that creation timestamp), and filter by the exact session id. If only an activity or updated date is known, or the date is unknown, first use a bounded metadata-only exact-thread or session-index lookup to derive the creation date, then scan that creation-date directory. If such a lookup is unavailable, ask for or derive the creation date from adjacent metadata; never substitute activity-date directories or widen `read_thread`.
- `session-meta` reads its prefix through raw descriptor calls so buffered prefetch cannot cross the exact byte cap. It accepts only complete LF-terminated JSONL records within that prefix; CRLF passes because it still ends in LF, while a bare CR is incomplete. It fails closed when the cap cuts through a record, even when the bytes through the cap form valid JSON, and when descriptor-relative directory enumeration is unreadable. Treat that explicit rollout error as a coverage gap; never reinterpret the missing row as proof that the rollout has no session metadata. The requested `--limit` bounds result rows only. Active prefix-proof work has an independent fixed safety cap of 501 consumed candidates (`MAX_SESSION_META_LIMIT + 1`); candidates with no `session_meta` consume that safety budget but do not consume the result-row allowance. Exhausting it emits `session_meta_candidate_limit_truncated` with `candidate_limit=501`: narrow the date or host scope before drawing a coverage conclusion, and do not treat a higher `--limit` as the remedy. A true result-row overflow remains `session_meta_limit_truncated` and may be addressed by narrowing the scope or raising `--limit` up to 500.
- During `session-meta` enumeration, filter the filename and requested date before treating `scandir` entries as name-only discovery; do not assume a cached dirent inode matches a fresh stat or that a child device matches its parent directory device. After filtering, capture each scoped name's raw metadata identity with a fresh descriptor-relative no-follow stat, then carry that inventory snapshot to consumption. The raw identity can represent symlinks and non-regular entries, so their precise errors stay deferred until consumption and a FIFO is classified without opening it. Do no rollout-open or prefix-proof I/O for unconsumed names. At actual consumption, securely open a selected regular name through the pinned parent with no-follow and nonblocking flags. Flat and date-nested `archived_sessions/**` candidates bind a full descriptor identity to fresh descriptor-relative no-follow stats and require the exact inventory snapshot across open, read, and final verification. Active `sessions/**` candidates also require the exact inventory identity, including size and timestamps, through the pre-open path check, opened descriptor, and first bounded prefix-proof read. Recheck that exact identity on the descriptor and a fresh descriptor-relative no-follow path snapshot after the proof read before treating the proof as the content baseline. Reject any append or grow-and-rewrite before this anchor as a retryable coverage gap because portable metadata cannot distinguish them; retry after the active rollout stabilizes. Capture that proof lazily for at most 501 consumed active candidates, then immediately re-read and extend it through the append-only checkpoint before parsing. Every later active checkpoint requires the same device and inode, forbids a size rollback below the latest observed high-water mark, and verifies that the proved prefix digest is unchanged; it keeps that high-water identity separate from the aligned verified-snapshot identity, and the parser reads only the immutable verified snapshot. If that snapshot has no `session_meta` and the post-parse checkpoint produces a newer aligned snapshot, parse the refreshed snapshot once and checkpoint again. If no metadata was found and the aligned snapshot advances again, or the returned high-water identity outpaces the latest aligned snapshot, fail with the existing explicit session-meta scan-truncated coverage error instead of silently skipping the rollout. Once the baseline exists, accept safe append growth while rejecting rewrite, truncation, replacement, rollback, and unbounded retry loops.
- After locating the rollout, treat `rollout-summary` and `chunked-rollout-summary` as locator and triage output only. They may retain only a representative or last user record from a chunk and cannot prove that every later substantive human follow-up was preserved. For full task reconstruction, never use summary rows to decide which chunks to omit. Before any chunk fetch, iterate every `chunk_meta` row in byte order and validate the complete plan: every row must report the same `source_bytes` and `full_fetch_limit_bytes`, the globally ordered advertised ranges must cover exactly byte 0 through `source_bytes` without gaps or overlaps, total planned bytes must equal `source_bytes`, and every row must report `full_reconstruction_allowed=true`. The automatic path additionally requires `source_bytes` at most 16 MiB (16,777,216 bytes); an over-limit path instead requires Joey's explicit authorization for that exact rollout and byte count. Before that authorization, fetch nothing; report the exact rollout, source byte count, and limit, or use a future server-side lossless filter. Never silently loop over an over-limit plan. After explicit authorization, track cumulative exact range bytes, preserve gap-free non-overlapping coverage, and stop at the authorized `source_bytes`; abort if the source or plan changes. Otherwise, fetch every listed `fetch_ranges[]` entry or the row's full `byte_start`/`byte_end` range with sequential bounded `fetch-rollout-chunk` calls, then hand the complete exact JSONL record stream to `codex-session-mining`. Do not batch multiple thread reads or stringify a raw result. Follow [Codex Thread Locator Skim](references/hosts.md#codex-thread-locator-skim) for the complete fallback and human-follow-up contract.
- Start every chunked full-reconstruction attempt with metadata-only `rollout-stat`. Record its exact `source_bytes` and versioned `source_identity`; this preflight does not scan or hash rollout content. Pass both values unchanged to `chunked-rollout-summary` and every `fetch-rollout-chunk` call through `--expected-source-bytes` and `--expected-source-identity`.
- The automatic source-read budget is 16 MiB. `chunked-rollout-summary` and `fetch-rollout-chunk` reject an over-limit identity before content scanning unless Joey authorized that exact rollout and exact byte count and the caller supplies the same value as `--authorized-source-bytes`. An authorization for any other size does not transfer, and a new identity requires a new decision.
- `chunked-rollout-summary` accepts chunks from 64 KiB through 2 MiB, caps each in-memory chunk at 4,096 physical records, and buffers no more than 4 MiB of final serialized JSONL summary output. A malformed JSONL record makes that chunk partial, increments `json_error_count`, and requires raw fetch even when other user and assistant evidence parsed successfully. Before allocating any explicit `fetch_ranges` list, and again while accumulating both explicit ranges and implicit whole-chunk fallback entries across all chunks, it rejects a plan above the advertised global `fetch_range_plan_limit` (currently 4,096 entries, derived conservatively from the summary-output budget). It scans exactly the preflight size from one identity-checked file descriptor, computes SHA-256 during that same allowed scan, and emits the digest in `rollout_meta` only after its post-scan descriptor and pinned-parent current-entry checks pass. Remote metadata and chunk-summary subprocess output is also captured with bounded stdout and stderr readers.
- Local and remote `session-meta` accept only complete serialized rows at or below 64 KiB; the remote parent caps complete stdout at 32,899,072 bytes. Remote `rollout-summary` enforces its producer-side serialized budget and caps parent stdout at 31,462,656 bytes. Either capture breach terminates the subprocess and fails before any partial frame is parsed or printed as command output; oversized metadata returns a complete fixed error rather than silently truncating locator fields.
- After fetching the complete ordered range plan, verify that the reconstructed byte count equals `source_bytes` and its SHA-256 equals `rollout_meta.source_sha256`, then run `rollout-stat` again with the original expected byte count and identity. Any append, truncation, replacement, short read, identity mismatch, digest mismatch, or final-stat mismatch invalidates the snapshot: discard the partial reconstruction and restart from byte 0 with a new `rollout-stat`; never extend the old plan or authorization.
- The helper takes a repeatable `--host` option for `preflight` and `session-meta`; do not pass positional host names or a plural `--hosts` flag.
- Treat active `sessions/YYYY/MM/DD/rollout-*.jsonl`, flat `archived_sessions/rollout-*.jsonl`, and date-nested `archived_sessions/YYYY/MM/DD/rollout-*.jsonl` paths as distinct locators even when their basenames and parsed session ids match. Deduplicate only an exact relative rollout path; lifecycle copies can contain different suffixes or later human follow-ups, so a repeated session id must not hide another path.
- For the default evidence scope, use `python3 /Users/hoteng/.codex/skills/remote-host-context/scripts/remote_codex_probe.py preflight --host local --host BL-mac-mini-m4-hoteng --host miku-bot-dev --host hoteng-srv-01 --host codex-hoteng-srv-01`.
- For required remote evidence workflows, if `remote_codex_probe.py preflight`, `session-meta`, `rollout-stat`, `rollout-summary`, `chunked-rollout-summary`, `fetch-rollout`, or `fetch-rollout-chunk` fails with a local sandbox/network error such as `Operation not permitted`, a DNS/network gate, or a connection blocked before SSH authentication, immediately rerun the same helper command with `sandbox_permissions=require_escalated`.
- Only report a host as unreachable after that escalated retry also fails, and include the exact gate from the failed retry.
- Use `... remote_codex_probe.py session-meta ...` only to list candidate session ids, cwd values, and canonical rollout paths from bounded `sessions/YYYY/MM/DD/` date trees.
- `session-meta` expects `--date YYYY/MM/DD`; do not pass ISO `YYYY-MM-DD`.
- Use `... remote_codex_probe.py fetch-rollout ...` only to copy one validated rollout file under task-scoped `.codex-tmp/remote-host-context/` beneath the current workspace, or under `/tmp`.
- `fetch-rollout` writes to a single file path via `--output <file>`; do not invent directory flags such as `--output-dir`.
- Local and embedded rollout readers inspect the raw expanded Codex-root entry with `lstat`, reject a final-entry symlink, open that same configured path directly with `O_DIRECTORY`/`O_NOFOLLOW`, and require the resulting `fstat` device and inode to match the inspection. They then pin every child ancestor with descriptor-relative `O_DIRECTORY`/`O_NOFOLLOW` traversal before opening the final regular file through the pinned parent with `O_NOFOLLOW` and `O_NONBLOCK`. The nonblocking final open prevents a stat-to-open FIFO replacement from hanging before the regular-file `fstat` rejection. They keep that parent descriptor through the read and revalidate the descriptor plus its current entry without retraversing a mutable pathname; missing required POSIX descriptor or nonblocking primitives fail closed.
- Local and embedded `fetch-rollout` producers freeze the opened descriptor identity, read no more than the snapshot size plus one growth-detection byte, require the returned length to equal the frozen size, and revalidate both descriptor and the final entry through its pinned parent after the read. The remote parent bounds the complete base64 frame before parsing it; `fetch-rollout-chunk` uses its own exact base64-frame stdout cap derived from the 2 MiB chunk limit. Any append, replacement, short read, or output-cap breach must fail before the destination is written.
- Local fetch output stays under the accepted task or system-temporary root. The helper opens or creates each parent through descriptor-relative `O_DIRECTORY`/`O_NOFOLLOW` traversal, then creates the private temporary file and performs the final rename relative to the same pinned parent descriptor; a parent symlink swap must fail or remain confined to that descriptor.
- When a validated rollout is too large to copy cleanly, run `rollout-stat` first, then use `... remote_codex_probe.py chunked-rollout-summary ...` with that exact identity to scan the frozen file snapshot in JSONL-record chunks and return structured evidence with byte/record ranges.
- `chunked-rollout-summary` is the default large-rollout locator; it provides whole-rollout chunk coverage and bounded coordinates, not lossless semantic coverage. Its leading `rollout_meta` row exposes the frozen identity and SHA-256, and each `chunk_meta` row repeats the common `source_bytes`, `source_identity`, `full_fetch_limit_bytes`, and fail-closed `full_reconstruction_allowed` decision needed before any exact fetch.
- Oversized single JSONL records are not parsed in the summary path; the helper emits `chunk_meta` and bounded `fetch_ranges` so exact wording can be fetched explicitly without loading the whole record during summary.
- `rollout-summary` remains a bounded prefix skim. It derives `source_bytes` from the same descriptor it scans and revalidates that descriptor plus the final entry through its pinned parent before emitting success. Keyword selection checks the full normalized signal before display truncation but retains only a transient boolean and bounded signal-only output, never the unbounded hidden text. If its `scan_meta.scan_truncated` is `true`, treat it as candidate selection only and upgrade to `chunked-rollout-summary` before writing an activity or work-report conclusion.
- `rollout-summary` emits a `scan_meta` row. If `scan_truncated` is `true`, treat the result as partial evidence and surface a coverage gap; do not summarize it as a complete scan.
- `chunked-rollout-summary` emits `chunk_meta` rows. If a chunk has `raw_fetch_recommended=true`, use `... remote_codex_probe.py fetch-rollout-chunk ...` for the listed `fetch_ranges`; oversized JSONL records may require fetching multiple ranges and concatenating them locally for exact wording.
- `rollout-summary` and `chunked-rollout-summary` output text is signal-only, not raw prompt/tool output text. Use it only for coarse friction flags, coverage, and candidate selection; a summary may prioritize coarse triage, but it must never decide omissions for full task reconstruction.
- `fetch-rollout-chunk` writes one bounded byte range via `--byte-start`, `--byte-end`, `--expected-source-bytes`, `--expected-source-identity`, and `--output <file>`; add the exact `--authorized-source-bytes` only for an approved over-limit snapshot. For full task reconstruction, first apply the complete-plan and 16 MiB authorization gate above. When allowed, consume all `chunk_meta` rows regardless of `raw_fetch_recommended`: use every `fetch_ranges[]` entry when present, otherwise fetch the row's full `byte_start`/`byte_end`, preserve global byte order, verify continuous coverage from byte 0 through `source_bytes`, verify the reconstructed SHA-256 and final stat, and pass the complete exact record stream to `codex-session-mining`.
- `fetch-rollout` may materialize an explicitly verified `archived_sessions/rollout-*.jsonl` path, but the helper should not widen `session-meta` into an unbounded archived-session crawler.
- Keep the helper focused on remote access and bounded file transfer. Do not turn it into a generic remote search shell.

3. Narrow the evidence before reading widely.
- For activity reports, inspect recent `~/.codex/sessions` trees and date-bounded rollout files first.
- Once a canonical remote rollout has been copied locally, prefer local filtering there over adding another remote `rg` command shape.
- Reserve bare remote `rg` for genuinely one-off preflight/debug checks that the helper cannot yet express, and patch the helper if that shape starts repeating.
- For multi-pattern remote searches, prefer repeated `-e` or fixed-string `-F` forms over a single shell-exposed regex such as `foo|bar`; if quoting would get brittle, stream the file back and filter locally instead.
- For repo-specific questions, inspect repo journals, worktrees, or nearby paths only after `~/.codex` indicates that host was actually active for the requested period.
- Keep remote reads bounded by date, repo, or task. Do not turn this into an unbounded home-directory crawl.

4. Delegate deeper rollout mining back to `codex-session-mining`.
- Once `fetch-rollout` has materialized the canonical remote rollout locally, let [$codex-session-mining](../codex-session-mining/SKILL.md) own the extraction, filtering, wrapper-noise skipping, and skill-friction classification.
- If `fetch-rollout` is blocked only by rollout size, use `chunked-rollout-summary` to enumerate the whole rollout, apply the complete-plan and 16 MiB authorization gate before fetching anything, then fetch every allowed range from every `chunk_meta` row in byte order and delegate the complete exact record stream to `codex-session-mining`; do not let lossy summary content choose omissions.
- Do not keep a second remote-only search flow here that duplicates `codex-session-mining` semantics.
- If the helper lacks one bounded remote-read primitive that keeps recurring, patch the helper or its references instead of approving another host-specific `ssh ... jq ...` literal.

5. Interpret host-specific structure correctly.
- `BL-mac-mini-m4-hoteng` currently stores Codex history directly in `/Users/hoteng/.codex`.
- `miku-bot-dev` currently stores Codex history directly in `/home/hoteng/.codex`.
- `hoteng-srv-01` currently stores Codex history directly in `/home/hoteng/.codex`.
- `codex-hoteng-srv-01` logs in as user `codex` on host `hoteng-srv-01` and stores Codex history directly in `/home/codex/.codex`; keep it in the default evidence scope because it is distinct from the `hoteng` account's history.
- On `hoteng-srv-01`, dev-shell-kit containers may mount the host `~/.codex` into the container. Treat the host path as canonical by default instead of entering containers first.
- If a host is stale relative to the requested date range, say that explicitly and deprioritize it instead of silently dropping it.

6. Feed the result into the parent workflow.
- For `$apple-notes-work-report`, merge remote host evidence before deciding an item is missing from the report.
- For `$apple-notes-work-report`, a truncated or raw-fetch-recommended remote rollout is an evidence gap until `chunked-rollout-summary` and any necessary bounded chunk fetches have been handled or explicitly blocked.
- For session-mining or workflow-summary tasks, list which hosts contributed evidence and which hosts were stale or unavailable.
- For repo recovery tasks, read remote repo journals only after the host preflight confirms the host was active.
- If helper-level bounded reads are still insufficient and the remote host already has Codex installed, a remote read-only Codex agent is an acceptable last-resort fallback. Scope it to one verified rollout or one repo/day question, require citations to session ids or rollout paths, and keep it read-only on the remote host.

## Guardrails

- Keep remote work read-only unless Joey explicitly authorizes modification.
- Do not silently fall back to local-only evidence when the user expects cross-host coverage.
- Do not enter `hoteng-srv-01` containers by default just because they exist; host-level `~/.codex` is the first source of truth.
- Do not collapse `hoteng-srv-01` and `codex-hoteng-srv-01` by remote hostname; both account-specific Codex roots belong in the default preflight.
- When SSH preflight fails, stop at the smallest useful decision point instead of guessing what happened on the remote host.
- When `remote_codex_probe.py` already covers the repeated host read, do not regress to host-specific bare `ssh`, remote `jq`, or remote `rg` literals just because they seem faster in the moment.
